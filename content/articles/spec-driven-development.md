很多人一听到Spec-Driven Development，第一反应是：这不还是写文档吗？

产品经理写PRD，研发看文档、拆需求、做设计、写代码。哪怕你把它叫Spec，本质不还是一样？

**这个质疑是对的。**

如果Spec只是"更详细的PRD"，那它和传统开发流程确实没有本质区别。研发仍然要从头理解需求，仍然要拆功能，仍然要判断接口、数据、异常和边界。

但真正的变化不在于多了一份文档。

**而在于AI介入后，产品经理和研发对齐的对象变了。**

过去对齐的是"需求理解"。现在对齐的是"可执行上下文"。

## 核心结论

传统PRD主要回答三个问题：为什么要做？给谁用？要做什么功能？

比如说，一个需求写成：

> 用户上传客户会议录音后，系统自动生成需求分析文档，并匹配历史项目，辅助产品经理快速判断项目可行性。

这句话对业务方来说够清楚，对产品评审也能讲通。

但研发拿到以后，仍然要继续追问——录音支持什么格式？多长录音算可处理？ASR失败怎么办？需求分析文档包含哪些字段？历史项目怎么匹配？相似度阈值是多少？AI生成结果能不能自动进入下一流程？哪些地方必须人工确认？有没有样例输入和样例输出？验收标准是什么？

**所以传统PRD的问题不是"不重要"，而是它通常停在业务描述层。**

研发真正开始做工程时，还需要做一次二次翻译：把业务语言翻译成接口、数据、状态、任务、异常、测试和上线边界。

这个二次翻译，在传统开发中靠人的经验完成。有经验的研发能猜、能追问、能补缺。但这个环节恰恰是AI介入后最大的断裂点。

## 二、AI介入后，模糊上下文成了致命伤

以前研发做二次翻译，主要是人来做。人有经验、有上下文、能追问、能猜。

**AI Coding介入后，二次翻译的一部分交给了AI Agent。**

问题是，AI最怕上下文模糊。

你只给它一句"自动生成需求分析文档"，它可能能写出一个Demo，但很容易在这些地方跑偏——自动确认了不该确认的需求；没有保留人工审核节点；生成内容没有引用来源；忽略权限和客户隔离；异常状态没有处理。

Demo看起来能跑，但不可上线。

**这不是AI笨，是你给它的上下文不够。**

AI就像一个能力很强但完全不了解业务的新同事。你跟他说"帮我做个需求分析功能"，他能做出来——但大概率不是你想要的那个。

那么，AI时代的产品研发协作应该怎么变？

不是产品经理写完PRD后让研发"自己理解"，也不是让AI"自动完成开发"。

**产品经理和研发要共同把模糊需求约束成AI和工程都能执行的上下文。**

这就是Spec的价值。

## 三、Spec不是更长的PRD，而是可执行的工程上下文

Spec不是一个更高级的PRD，也不是研发详细设计的替代品。

我觉得更准确的说法是：

**Spec是介于PRD和工程实现之间的一套结构化任务说明，它把业务需求转成研发和AI Coding工具都能使用的工程上下文。**

它通常包含这些内容：

| 维度 | 传统PRD常见写法 | Spec需要补充 |
|------|---------------|-------------|
| 业务目标 | 说明为什么做 | 明确成功标准和不做什么 |
| 用户流程 | 描述主流程 | 补状态、权限、异常和兜底 |
| 功能范围 | 列功能点 | 拆成可开发任务 |
| 数据 | 说需要哪些信息 | 明确字段、来源、读写规则 |
| 接口 | 可能只写对接系统 | 明确入参、出参、错误码 |
| AI能力 | 写"智能生成" | 明确输入、输出、置信度、人工确认 |
| 验收 | 写功能可用 | 给测试样例和通过标准 |

**所以Spec的核心不是"写得更长"，而是"写得更可执行"。**

## 四、一个Spec长什么样

还是刚才那个"会议录音生成需求分析文档"的需求。

PRD可能写：

> 用户上传客户会议录音后，系统自动生成需求分析文档，并匹配历史项目，辅助产品经理快速判断项目可行性。

Spec会进一步拆成：

```markdown
功能：会议录音生成需求分析文档

输入：
- audio_file
- customer_id
- project_type
- uploader_user_id

处理流程：
1. 调用ASR服务，将录音转为文字
2. 抽取客户背景、业务痛点、功能需求、预算、时间、约束
3. 调用历史项目知识库，检索相似项目
4. 相似度高于阈值时，输出可复用方案建议
5. 相似度不足时，生成新需求分析初稿
6. 所有结果进入人工确认节点，不自动进入开发池

AI能力边界：
- AI只能生成初稿
- 不允许自动确认客户需求
- 不允许自动承诺交付周期
- 不允许自动生成报价

异常处理：
- ASR失败：提示重新上传
- 知识库无匹配：走新需求分析
- 置信度低：标记"需人工确认"

验收标准：
- 30分钟以内录音可在5分钟内生成初稿
- 文档必须包含来源引用
- 进入下一流程前必须有人确认
```

这仍然是文档。但它已经不是"需求说明"那么简单了。

**它把研发最容易反复追问的问题提前结构化了，也给AI Coding工具提供了明确约束。**

输入是什么、输出是什么、哪些能自动化、哪些不能自动化、失败时怎么办、做到什么算完成——全部写清楚了。

## 五、研发是不是还要拆需求

当然要。

Spec不是替代研发拆需求。

研发依然要做：技术方案设计、前后端任务拆分、数据库和接口设计、模型/RAG/Agent调用方案、性能/安全/权限/部署判断、代码Review和质量把关。

区别在于，研发不再是从一段模糊PRD里重新猜业务边界，而是在一个更清晰的上下文上做工程判断。

传统流程里，研发经常在开发过程中不断问产品：这个字段必填吗？这个状态怎么处理？AI结果错了怎么办？这个流程能不能自动进入下一步？

**Spec的作用，是把这些高频问题前置。它不能消灭沟通，但能减少低质量沟通。**

## 六、AI时代产品研发对齐的五个变化

### 1. 产品经理要定义AI的能力边界

过去产品经理主要定义功能。

现在还要定义：AI能做什么、AI不能做什么、哪些结果必须引用来源、哪些场景必须人工确认、哪些任务不能自动进入下一流程、低置信度时怎么兜底。

**这些边界如果产品不提前说清，研发和AI Agent很容易按"能自动就自动"的方向做，最后带来业务和合规风险。**

### 2. 产品经理要提供样例，而不只是描述

AI Coding很依赖样例。所以产品经理不能只写"生成一份需求分析文档"。

更应该提供：输入样例、输出样例、正确样例、错误样例、Badcase、验收样例。

**样例会直接影响研发怎么测试，也影响AI Agent怎么生成结果。**

### 3. 研发拆的不只是功能，还有AI链路

传统研发拆的是页面、接口、数据库、前后端任务。

AI项目里还要拆：Prompt、RAG知识库、Agent工作流、工具调用、权限控制、置信度判断、人工确认节点、日志和审计、Badcase回流。

**产品和研发对齐时，不能只讲页面和功能，还要讲AI链路怎么被约束、怎么被评估、怎么失败兜底。**

### 4. 对齐方式从一次性交接变成持续校准

传统模式是：PRD评审 → 研发排期 → 开发 → 测试 → 上线。

AI Coding之后，更像：Spec初版 → 快速Demo → 业务验收 → 发现Badcase → 补Spec/Rules/测试样例 → 再迭代。

**产品和研发不只是交接关系，而是在共同维护一套规则。**

这套规则包括：需求边界、AI输出边界、工具调用边界、质量标准、Badcase处理方式。

### 5. 验收从"功能可用"变成"结果可控"

传统验收看页面是否可用、流程是否走通、数据是否正确、权限是否生效。

AI项目还要看：回答是否有依据、召回是否命中、生成是否幻觉、置信度低时是否兜底、是否越权调用工具、Token成本是否可控、Badcase是否能回流。

**产品经理不只是验收功能，还要参与AI质量标准的定义。**

## 七、产品经理的新角色：上下文设计师

回到最开始的问题——AI时代，产品经理的价值在哪里？

不是"写更多文档"。

**而是把业务、研发和AI之间的不确定性提前收敛。**

具体来说，产品经理要做三件事——

第一，把模糊业务需求转成明确任务。

第二，把AI能力边界转成可执行规则。

第三，把上线后的Badcase转成下一轮迭代资产。

产品经理更像一个"上下文设计师"。

他不是替研发写代码，也不是替业务拍脑袋，而是在业务目标、工程实现和AI能力之间设计一套可执行、可校验、可迭代的上下文。

## 八、Spec与SDD的关系

Spec-Driven Development（SDD）不是新概念，但在AI时代有了新的含义。

传统SDD强调"先写规格再写代码"——规格满足即终止。但在AI时代，Spec的角色发生了变化：

**Spec不只是代码的输入，它是AI Agent的运行约束。**

- Spec定义了AI能做什么、不能做什么 → AI能力边界
- Spec定义了输入输出格式 → Agent调用的契约
- Spec定义了异常处理 → Agent的兜底逻辑
- Spec定义了验收标准 → 验证器的判断依据
- Spec定义了人工确认节点 → Human-in-the-loop的介入点

也就是说，Spec从"人和人之间的对齐工具"变成了"人、AI和工程系统之间的共享上下文"。

传统PRD是告诉研发：要做什么。

**AI时代的Spec是告诉研发和AI Agent：在什么边界内，按什么规则，把什么输入变成什么输出，并用什么标准验收。**

## 九、Spec的行业实践与工具生态

Spec驱动开发在2026年已经不是纸上谈兵。GitHub的spec-kit、Anthropic的Claude Code工作流、各家Agent框架的schema约束机制，都在往这个方向走。

几个值得关注的实践方向：

**1. Spec即代码**

把Spec写成结构化文件（YAML/JSON），而不是自然语言文档。好处是AI Agent可以直接解析、执行、校验。spec-kit就是这个思路——Spec本身是可执行的。

**2. Spec → Agent → Spec的迭代闭环**

Spec定义初始约束 → AI Agent执行 → 发现偏差 → 更新Spec → 再执行。这本质上是Loop Engineering在需求层面的应用。Spec既是输入，也是迭代的产出物。

**3. Spec与RAG的结合**

把历史Spec、Badcase、验收样例沉淀成知识库，新需求来时自动检索相似Spec做参考。这比从零写Spec快得多，也减少了遗漏。

**4. Spec的版本管理**

Spec不是一次写完的，它随项目演进。把Spec纳入版本管理，每次迭代都有diff，能追溯"为什么改了这条约束"。这比传统PRD的版本管理更有工程意义。

回到最开始所说的——AI时代产品研发协作的范式迁移。

它不是魔法，也不是让研发不用拆需求。

它只是把过去大量隐性的沟通、追问、返工和上下文补充，尽可能前置成一套结构化材料。

AI介入以后，产品经理和研发的关系不会变得更轻。相反，它会要求双方更早、更细、更持续地对齐：业务目标、工程边界、AI能力、质量标准和风险兜底。

**产品经理不再只是需求的描述者，而要成为AI时代工程上下文的组织者。**
