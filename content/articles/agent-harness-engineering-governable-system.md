## 核心结论

很多人把"做 Agent"等同于"调模型 API + 写 Prompt"。但模型本身只是一个根据输入给出输出的函数，它不天然理解业务状态、历史轨迹和运行边界。

真正让 Agent 在实际场景中稳定工作的，是模型外部的一整套驾驭系统——**Harness**。

**模型**负责语言理解、推理和生成。**Harness** 负责一切工程化事务：

- 提示词管理与上下文构建
- 工具调用流程编排
- 运行状态与 Memory 管理
- Skills / MCP / 插件的加载与调度
- 运行轨迹记录与复盘
- 数据治理与评估反馈

可以把 Harness 理解为模型的"驾驭工程"：它决定模型在什么场景下使用什么工具、读取什么上下文、遵循什么业务边界、如何记录过程、如何复盘失败，以及如何把运行数据转化为后续优化所需的数据资产。

## 二、传统 Agent 落地的五个痛点

理解 Harness 的价值，先要看清不做 Harness 工程化时会遇到什么：

1. **运行记录分散**——对话记录、工具调用日志、Memory 文件、调试信息散落在文件系统或不同日志平台，后续难以统一查询和复盘。

2. **上下文不可治理**——Agent 的上下文不仅包含用户输入，还包含工具调用过程、状态变化、失败原因、阶段性结论等运行事实。如果这些信息只以普通聊天记录存在，就无法被 Agent 自身再次利用。

3. **工具与业务逻辑耦合严重**——过多能力直接内置到主流程中，系统越来越重，某个能力出问题可能影响整体稳定性。

4. **快速验证成本高**——自行部署模型服务、MCP Server、数据库和前端应用，工程成本高，不利于快速验证一个 Agent 场景是否成立。

5. **缺少数据闭环**——运行过程中产生的大量数据如果不能沉淀为评估数据、调试数据和训练数据，就无法反哺下一轮优化。

> 本质问题：如何用较低成本搭建一个 Agent 系统，使运行数据从第一天起就具备可查询、可观测、可复盘和可持续优化的能力。

## 三、架构设计：轻内核 + 插件化扩展

### 3.1 轻内核的职责边界

Agent 框架不应该把所有能力都内置到核心流程里。核心原则是：**维护一个稳定、清晰、轻量的内核，通过插件扩展业务能力。**

**内核只负责：**

- Agent Loop 的稳定运行
- 会话和状态管理
- 输入输出通道抽象
- 工具调用流程编排
- 上下文构建
- 运行轨迹记录

**插件负责扩展：**

- MCP 工具能力
- Memory 能力
- IM 通道（飞书、微信等）
- 定时任务
- 可视化界面
- 数据库适配
- 业务专属 Skills

这种设计的好处：某个插件出现问题时，可以局部摘除或替换，不会直接破坏整个 Agent 系统。对于生产环境来说，这比将所有能力捆绑在主流程中更容易维护。

### 3.2 阶段化拆分交互流程

Agent 的一次交互不应该被看成单一的"输入-输出"，而应拆成多个明确阶段：

1. 接收用户输入
2. 识别任务意图
3. 构建上下文
4. 选择工具或 Skills
5. 调用工具
6. 记录运行事实
7. 生成响应
8. 写入运行轨迹
9. 必要时进行复盘或沉默

阶段化拆分让工程团队可以在不同阶段挂载 Hook，实现细粒度扩展。例如在上下文构建阶段注入 Memory，在工具调用阶段接入 MCP，在输出阶段决定是否回复用户。

**一个关键经验：Agent 不一定每次都要回复。** 在多人协作或多 Agent 协作场景中，过多回复会变成噪音。输入通道和输出通道不应被强绑定，Agent 应允许在某些场景下保持沉默。

### 3.3 Skills 与 Tools 的边界

实践中需要区分三个层次的能力扩展：

| 层次 | 定位 | 示例 |
|------|------|------|
| **Tools / MCP** | 可调用的单点工具 | 时间服务、地图服务、检索服务 |
| **Skills** | 标准化 SOP，沉淀任务方法论 | 文档规范、代码审查方法、工作流 |
| **插件** | 大粒度能力扩展，可同时包含工具、Skills、通道和运行时逻辑 | IM 通道插件、数据库适配插件 |

Skills 的价值在于**减少 Agent Loop 的复杂度**。不是所有能力都应该写进核心流程，有些通用经验可以作为可分发、可复用的技能包存在。

## 四、数据闭环：从运行事实到自我复盘

### 4.1 Tap：运行事实而非普通聊天记录

传统聊天记录只保存"用户说了什么、Agent 回了什么"。但如果要支撑排查和复盘，需要记录的是**运行事实（Tap）**——对任务有意义的关键过程：

- 用户提出了什么问题
- Agent 识别出了什么任务
- 使用了哪些工具
- 工具返回了什么关键结果
- 哪一步失败或中断
- Agent 做出了什么判断
- 最终输出是什么

Tap 与传统可观测性数据相比更简洁，但它有一个关键优势：**既能给人看，也能给 Agent 自己看。** 工程师和 Agent 可以基于同一份运行事实进行排查和复盘。

### 4.2 从可观测性到自我复盘

过去排查 Agent 问题时，工程师需要进入日志平台，从 Trace、日志和指标中人工定位问题。

在 Harness 的设计中，**Agent 的运行轨迹本身就是上下文的一部分。** 用户或工程师可以直接问 Agent：

- 刚才为什么没有回答成功？
- 哪个工具调用失败了？
- 这次回答用了哪些上下文？
- 这个任务和上一次任务有什么差异？

Agent 可以读取自己的运行事实，对失败原因进行初步分析。这种设计将一部分工程排查能力前移到了 Agent 自身。

### 4.3 数据资产化

当运行轨迹被结构化记录后，可以形成完整的数据闭环：

- 对比不同模型在同类任务上的表现
- 评估同一模型在多次运行中的稳定性
- 沉淀失败案例和边界案例
- 脱敏后导出为调试数据集
- 用于后续模型微调、评测和训练

## 五、数据库原生 Harness：为什么不用文件系统

很多 Agent 项目依赖文件系统管理 Memory（Markdown 文件、日志文件、本地缓存）。这种方式对人类阅读友好，但不适合做结构化查询、统一治理和长期数据闭环。

**数据库原生 Harness 的核心优势：**

- 所有运行数据统一存储
- 支持结构化查询和分析
- 支持上下文、Memory、Trace、任务记录之间的关联
- 便于做权限管理、归档和脱敏
- 便于沉淀评估数据集
- 可以支撑从本地验证到生产规模的演进

数据库不仅是业务数据存储组件，也是 Agent Harness 的运行底座。它让 Agent 的上下文、运行轨迹和评估数据从一开始就具备可治理能力。

## 六、Agent 系统评估维度

评估一个 Agent 系统是否可用，不能只看回答是否"看起来正确"，还需要关注过程质量和工程可控性：

| 维度 | 核心问题 |
|------|----------|
| **任务完成率** | Agent 是否能完成用户提出的核心任务 |
| **工具调用成功率** | MCP 或其他工具是否被正确选择、正确调用，并返回可用结果 |
| **工具调用轮数** | 完成简单任务是否需要过多轮调用（过多说明规划或选择有问题） |
| **上下文利用质量** | Agent 是否正确读取了历史运行事实、Memory 或数据库中的相关信息 |
| **失败可解释性** | 任务失败时，能否定位失败发生在哪一步，能否说明失败原因 |
| **扩展成本** | 新增 MCP、Skill 或插件时，是否需要改动核心 Agent Loop |
| **数据闭环能力** | 运行数据是否能被查询、分析、脱敏导出，并用于后续优化 |

## 七、实践中的关键经验

### 7.1 先跑通最小闭环，再扩展能力

最小可用链路：

- 模型 API 可用
- 数据库可用
- 对话界面可访问
- Agent 能完成一次基础问答
- 运行记录可查询

在此基础上再逐步添加 MCP、Skills 和业务插件。这样可以降低问题定位成本。

### 7.2 工具和技能不要一次性加太多

工具和技能越多，Agent 的选择空间越大，也越容易出现错误调用或上下文膨胀。建议按任务场景逐个添加，确认可用后再扩展。

### 7.3 把运行轨迹当成产品能力

Agent 的运行轨迹不只是调试日志，而是用户信任系统的重要依据。能解释自己为什么这样回答、为什么失败、用了哪些工具，是面向真实业务落地的关键能力。

### 7.4 数据库是 Agent 工程化的基础设施

当 Agent 从 Demo 走向生产时，数据管理会成为核心问题。将上下文、Memory、工具调用、任务轨迹和评估数据统一纳入数据库管理，可以显著降低后续治理成本。

### 7.5 Docker 适合快速验证，源码部署适合二次开发

如果目标是快速体验，Docker 一键部署更合适。如果要深入修改框架、扩展插件或调试底层逻辑，则应考虑本地源码部署。

## 八、方法论总结

这套实践验证了一个面向生产落地的 Agent Harness 思路：

> 模型只是智能体能力的一部分，真正决定 Agent 是否可持续运行、可治理、可扩展的，是模型外部的 Harness。

核心工程闭环：

- **用数据库统一承载 Agent 运行数据**
- **用 Tap 记录关键运行事实**（而非普通聊天记录）
- **用 MCP 和 Skills 扩展业务能力**（而非堆砌核心流程）
- **用插件化架构降低核心系统复杂度**（轻内核 + 可替换插件）
- **用运行轨迹支持调试、评估和训练反馈**（数据资产化）

这套方法论为构建更复杂的多 Agent 系统、多人协作 Agent 和业务问答助手提供了可复用的工程经验。
