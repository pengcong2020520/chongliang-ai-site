## 核心结论

传统RAG的工作流是：用户提问 → 直接用原始问题检索知识库 → 把检索到的片段塞进Prompt → LLM生成回答。

这个流程的瓶颈在第一步。用户的提问往往是口语化的、模糊的、术语不规范的。

比如说用户问"我想了解一下我们公司新出的那款笔记本电脑，最近好像有人说存不上文件，是怎么回事？"——如果你直接拿这段话去做向量检索，检索引擎会被大量无关信息干扰，召回的知识片段可能完全不对。

**问题的核心不在于检索引擎不好，而在于输入给检索引擎的query质量太低。**

## 二、解法：多关键词并行检索

核心思路：**不直接用用户query检索，而是让LLM先提取6-8个精准关键词，再用这些关键词分别检索知识库，最后整合结果生成回答。**

工作流程：

```
用户提问 → LLM提取关键词(6-8个) → 代码解析为数组 → 迭代检索知识库(每个关键词一次) → 整合检索结果 → LLM生成带引用的回答
```

### 关键词提取的四步逻辑

LLM提取关键词不是简单分词，而是做四件事：

1. **语义焦点识别**：从长文本问题中定位核心实体（产品名、服务名、错误代码），避免被次要信息干扰
2. **术语标准化**：口语化表达映射到知识库官方术语（"手提电脑"→"笔记本电脑"，"存不上文件"→"文件保存失败"）
3. **上下文增强**：补充隐含条件和领域知识（"最新版软件闪退"→["v2.3.1", "程序崩溃"]）
4. **消除歧义**：对多义词添加限定词，排除非相关实体

### 并行检索

多个关键词同时查询知识库，每个关键词返回Top-K条结果。迭代节点遍历关键词数组，逐条检索，收集所有结果。

**关键设计**：迭代检索时使用原始问题的完整语义做向量匹配，而关键词的作用是在前置环节优化问题表达、扩大检索覆盖范围。

### 结果整合与回答生成

LLM拿到所有检索结果后，做三件事：

1. **空值保护**：如果知识库无相关结果或置信度低于阈值，明确告知无法回答，禁止编造
2. **结论先行**：采用结论先行的回答结构，关键数据标注来源编号
3. **自然语言转化**：技术术语转为用户可理解表述，操作步骤添加风险提示

## 三、设计亮点

| 设计点 | 解决的问题 | 价值 |
|--------|-----------|------|
| 关键词扩展 | 用户表达口语化、术语不规范 | 提升检索精准度 |
| 并行检索 | 单次检索召回不足 | 扩大召回范围 |
| 结果整合 | 多路检索结果碎片化 | 统一生成连贯回答 |
| 来源追溯 | 回答不可验证 | 可关联到知识库具体段落 |
| 空值保护 | LLM幻觉 | 无结果时明确告知，禁止编造 |

## 四、潜在优化方向

1. **重排序启用**：配置了Rerank但未启用，开启后可进一步提升检索结果质量
2. **并行执行**：迭代的并行参数配置需根据实际需求调整
3. **置信度过滤**：设置分数阈值过滤低质量结果，减少噪声

## 五、方法论总结

多关键词并行检索的核心判断是：**RAG的瓶颈不在检索引擎，在输入质量。**

传统做法把精力花在调检索参数（Top-K、分数阈值、Rerank模型）上，但如果输入的query本身就是垃圾，再好的检索引擎也救不回来。

**先把输入质量搞上去——关键词扩展、术语标准化、消歧——再做检索优化，效果会好得多。**

这跟做产品是一样的道理：用户说出来的话和用户真正想要的往往不是一回事。你不翻译就直接执行，大概率跑偏。
