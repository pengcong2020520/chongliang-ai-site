## 核心结论

用Claude Code做项目，最痛的问题不是AI不够聪明，而是"失忆"。

对话太长，每轮消耗的token可能是前期的3-5倍——因为每发一条消息，AI都要把整个上下文重新处理一遍。所以精明的用户会定期开新对话来"重置"上下文长度，省钱。

但新对话=空白记忆。你得把项目背景、技术栈、当前任务、编码规范全部重新喂一遍。这又是一笔token开销。

**核心矛盾：长对话贵，短对话忘。你被夹在中间，左右为难。**

唯一的解法是：**让项目的关键信息持久化到文件里，AI每次开新对话自动读取，而不是靠你手动喂。**

## 二、最重要的技术判断：TODO不能写进CLAUDE.md

这是一个踩过最贵的坑。

一开始的逻辑是：CLAUDE.md是给AI看的"员工手册"，那把TODO也写进去不就行了？一个文件搞定，多省事。

**大错特错。**

要理解为什么，得先知道一件事：**CLAUDE.md在AI编程工具里的角色，相当于System Prompt。**

System Prompt有一个巨大的优势——**Prompt Caching**。

如果你的System Prompt每次都一样，API会缓存它的处理结果。下次调用时，不需要重新处理这部分内容，直接用缓存，省钱、省时间。

但如果你频繁修改CLAUDE.md（比如每完成一个任务就更新一下任务状态），缓存就会失效。每次调用都要重新处理整个CLAUDE.md，等于白白多花好几倍的钱。

**CLAUDE.md应该是"几乎不改"的。它写的是规则，不是状态。规则不变，缓存才有效。**

那动态的任务状态放哪？**单独放TODO.md。**

| 文件 | 修改频率 | 缓存影响 | 内容类型 |
|------|---------|---------|---------|
| CLAUDE.md | 极少修改（稳定规则） | ✅ 高缓存命中率 | 系统指令、编码规范 |
| TODO.md | 频繁修改（任务更新） | ❌ 不需要缓存 | 动态任务状态 |
| README.md | 极少修改（稳定概述） | ✅ 高缓存命中率 | 项目介绍、技术栈 |

**一句话总结：CLAUDE.md是宪法，改一次要全民公投；TODO.md是日程表，每天都可以划掉几项。别把日程表写进宪法里。**

## 三、三层上下文系统

搞清楚了文件分工，整个系统就清晰了：

| 层级 | 文件 | 解决的问题 | 面向对象 |
|------|------|-----------|---------|
| 项目级 | README.md | "这是什么项目？" | 人 + AI |
| 任务级 | TODO.md | "现在在做什么？" | 人 + AI |
| 对话级 | CLAUDE.md + 聊天指令 | "这一轮怎么配合？" | AI专用 |

- **README.md**是"公司简介"——让AI知道这是什么项目、用什么技术栈、有什么规矩
- **TODO.md**是"今日待办"——让AI知道当前做到哪了、下一步做什么、有哪些bug
- **CLAUDE.md**是"员工手册"——让AI知道写代码的规矩、测试要求、协作流程

三层各司其职，互不干扰。AI每次开新对话，自动读取这三个文件，立刻进入状态，不需要你重新介绍。

## 四、为什么这个设计有效：上下文持久化的经济学

这个三层文件系统不只是"方便"，它解决了一个经济学问题。

Vibe Coding的成本结构里，最大的变量是token消耗。而token消耗的最大来源不是你打字输入的那几句话——是AI每次都要重新处理的上下文。

**三层文件系统的精妙之处在于：它把上下文分成了"可缓存的"和"不可缓存的"两部分。**

可缓存的部分（CLAUDE.md + README.md）：
- 内容稳定，几乎不改
- 被AI作为System Prompt读取，享受Prompt Caching
- 缓存命中时，处理成本降低60-80%
- 你开100次新对话，这部分只付一次完整处理费

不可缓存的部分（TODO.md + 当前对话）：
- 内容动态，频繁更新
- 作为普通上下文读取，不走缓存
- 但内容量小（通常只有几十行），成本可控

**也就是说，三层文件系统不是随意的文件分类，是围绕Prompt Caching的经济学做的架构设计。**

## 五、四个即用模板

### 模板一：README.md

```markdown
# 项目名称

## 这是什么？
一句话说清楚：这是一个 ______，解决 ______ 问题。

## 技术栈
- 前端：框架 + 语言
- UI：组件库
- 后端：数据库 + 认证
- 部署：平台

## 目录结构
- app/：页面和路由
- components/：通用组件
- lib/：工具函数和配置

## 开发规范
- 必须使用TypeScript，禁止.js新文件
- 组件用函数式+hooks，不用class
- 错误处理必须完整，禁止空catch
```

### 模板二：TODO.md

```markdown
# TODO

## ✅ 已完成
- [x] 初始化项目结构
- [x] 配置数据库连接

## 🚧 进行中
- [ ] 实现用户登录页面

## 📋 待开发
- [ ] 用户注册页面
- [ ] 密码重置功能

## 🐛 已知问题
- [ ] 移动端按钮点击区域太小
```

### 模板三：CLAUDE.md

```markdown
# CLAUDE.md

## 你是谁
你是这个项目的高级工程师。职责：写代码、解释决策、维护文档。

## 开始前必读
1. 读README.md了解项目
2. 读TODO.md确认当前任务
3. 用自己的话复述任务目标
4. 给出实现步骤，确认后动手

## 代码规范
- TypeScript only，禁止any
- 函数式组件+hooks
- 不准引入新依赖，除非我同意
- 错误处理必须完整

## 完成后
1. 输出变更摘要
2. 提示验证命令
3. 更新TODO.md

## 禁止事项
- 不准换框架
- 不准删除安全相关代码
- 不准输出任何密钥
```

### 模板四：新对话起手Prompt

```
你是这个项目的高级工程师。

【项目背景】
[粘贴README.md的技术栈+目录结构+开发规范]

【当前任务】
[粘贴TODO.md中"进行中"的那一条]

开始前：
1. 复述你理解的任务目标和约束
2. 给出实现步骤
3. 确认后开始写代码
4. 完成后更新TODO.md
```

## 六、日常使用的四个习惯

模板只是起点，真正让系统跑起来的是日常习惯：

1. **一个对话只干一件事**——登录一个对话，支付新开一个，别贪心
2. **定期让它汇报工作**——对话中期让它总结进度，复制到新对话"重新培训"
3. **发现偷懒立即纠正**——引用CLAUDE.md规则让它重写，不纠正它会变本加厉
4. **让它自己维护文档**——完成后让AI更新TODO.md，保持状态同步

## 七、底层逻辑：上下文持久化的通用模式

三层文件系统的思路不只适用于Claude Code，它是一个通用模式。

任何"AI辅助长周期项目"都会面临同样的矛盾：长上下文贵，短上下文忘。解法都是一样的——**把上下文分成稳定的（可缓存）和动态的（不可缓存），用文件系统持久化，让AI自动读取。**

这个模式可以迁移到：
- **知识库管理**：README.md→知识库索引，TODO.md→当前研究任务，CLAUDE.md→加工规则
- **Agent编排**：项目级Spec + 任务级状态 + 对话级指令
- **团队协作**：项目文档 + 任务看板 + 协作规范

回到最开始所说的——Vibe Coding的上下文管理本质上就解决一个问题：**让AI在每次新对话里，用最少的token恢复到最佳工作状态。**

三个文件，三层上下文，四个习惯。

不复杂，但需要你真的去建、去维护。很多人的问题不是不知道这些道理，而是懒得建那三个md文件。

别懒。花10分钟建好，后面省的是无数轮重复解释的token和时间。
