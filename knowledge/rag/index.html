DEFINED TERM

RAG / 检索增强生成 是什么

RAG 是让大模型在生成回答前先检索外部知识库的技术，解决模型缺乏私有知识、知识无法实时更新、回答需要可追溯来源的问题。

LLM Wiki

定义：RAG 是一种将检索与生成结合的技术方法，通过离线阶段将文档解析、切块、Embedding 入库，线上阶段用用户查询检索相关片段并注入 Prompt，使大模型基于私有知识准确回答。

适用场景

  • 企业私有文档问答
  • 需求分析与历史项目匹配
  • 合规性溯源回答
  • 多Agent架构中专家Agent的知识挂载

它不是什么

它不等于"给模型加个搜索引擎"，也不等于微调——RAG 不改变模型参数，只改变模型在生成时能看到什么上下文；RAG 也不保证答案正确，需要全链路调优和质量验证。

常见误区

  • 认为 RAG 的瓶颈在检索引擎，实际上瓶颈在输入质量和切块策略
  • 忽视 chunk 大小调优——过大导致检索精度下降，过小导致上下文不完整
  • 只关注整体准确率而不做全链路追踪，无法定位具体哪个环节出了问题
  • 认为 Graph RAG 万能——它只在跨文本推理和复杂关系梳理场景才有明显提升

更完整的解释

RAG 是当前 Agent 工程化的三类基础设施技术之一（Prompt 工程、多Agent架构、RAG），它解决的核心问题是：仅靠 LLM 的参数化知识无法覆盖企业私有文档，必须通过检索机制将历史文档注入生成过程。

标准流程

离线处理：非结构化数据（PDF、图片等）→ OCR 预处理转 Markdown → 按策略切块 → Embedding 后存入向量数据库建立索引

线上查询：用户 Query 与向量库做相似度匹配（粗筛）→ Top-K 结果做重排序（精选）→ 抓取最终结果给大模型组装输出

核心调优：文本切块三层法

RAG 调优中最核心的环节：

1. 段落切分：按段落（\n\n）做初步切块 2. 语句切分：段落仍过长时，按标点符号切分为单个句子 3. 语义切分：通过语义脉冲分析做调整——脉冲平缓说明前后语义关联度高，重新拼接召回；脉冲明显说明前后语义差异大，再次切分

调优后整体文档处理准确率可从 80% 提升至 95% 以上。

全链路追踪

传统 RAG 的核心痛点：只知道整体准确率不高，无法定位是哪个环节出了问题。解法是引入 Run ID 串联整条链路，在切块、Embedding、向量匹配等每个环节都记录数据。当问答准确率不足时，通过 Run ID 快速定位具体问题模块。

多关键词并行检索

传统 RAG 直接用用户原始 query 检索，口语化表达导致召回率低。解法是让 LLM 先从问题中提取 6-8 个精准关键词，并行检索后整合结果。核心判断：RAG 的瓶颈不在检索引擎，在输入质量。 先做关键词扩展、术语标准化、消歧，再做检索优化，效果才好。

输出质量验证

强制溯源（输出必须带上来源索引）→ AI 自动判别引用关联性 → 评测集测试 → 小批量用户体验 → Bad Case 回流调优。

三类技术的定位关系

  • 多Agent架构是系统级组织方式，决定"谁来做"
  • RAG是知识级增强手段，决定"用什么知识做"
  • Prompt 工程是交互级优化手段，决定"怎么跟 LLM 说"

三者是嵌套互补关系，不是平级竞争。

例子

企业知识问答场景：用户问"我想了解一下我们公司新出的那款笔记本电脑，最近好像有人说存不上文件"——如果直接用这段话做向量检索，会被大量无关信息干扰。正确做法是让 LLM 先提取关键词（"笔记本电脑"、"文件保存失败"等），标准化术语后并行检索，最终生成带来源引用的回答。这就是 RAG 在企业落地中的典型工作流。
