DEFINED TERM

Agent 安全 是什么

Agent 安全解决的是 AI 智能体拥有自主决策权和工具调用能力后带来的新风险类别——权限悖论（权限太少干不了活，太多出生产事故）。核心是建立分级的、动态的、可审计的权限控制体系。

定义：Agent 安全是指在 AI Agent 具备自主感知、推理、决策和执行能力的背景下，围绕 Agent 的权限边界、行为可控性、数据安全性和合规性所展开的系统性安全治理框架。与传统应用安全的根本差异在于：Agent 拥有自主决策权——即使代码完全安全，AI 的"合理推理"仍可能导致危险行为。

适用场景

  • 智能体运维数据库时自动重启主机的权限悖论场景
  • 低权限用户通过提示注入诱导Agent泄露高权限数据
  • 多Agent协作中一端被攻陷后骗取其他Agent敏感信息
  • MCP工具调用参数被恶意注入执行危险操作

它不是什么

它不是传统应用安全的子集——传统安全关注代码漏洞，Agent安全还要解决AI判断本身的风险；不是Agent Harness（驾驭工程）——Harness偏工程化治理（让Agent跑得稳），Security偏风险合规（让Agent跑得安全）；不是AI对齐——对齐关注模型价值观，Agent安全关注运行时可工程化的安全控制。

常见误区

  • 认为Agent安全等于加个权限管理模块——实际是系统性工程涵盖沙箱/身份/审计/合规
  • 认为测试环境跑通生产就安全——Agent行为是概率性的同一输入可能不同输出
  • 认为用强模型就安全了——Agent能力越强越危险更容易被社会工程攻击
  • 认为沙箱就够了——沙箱不解决提示注入和合规问题
  • 忽视Security for AI只关注AI for Security——Agent自身才是最大攻击面

更完整的解释

权限悖论：Agent落地的第一障碍

企业落地Agent的核心障碍不是模型能力，而是权限悖论——权限太少干不了活，太多出生产事故。

真实案例：企业让AI Agent管理数据库实例，Agent检测到EC2主机系统漏洞后，"合理地"自动更新系统并重启主机，导致生产业务中断。Agent的推理链条完全正确（发现漏洞→修复漏洞），但缺乏权限边界约束时，正确推理也会导致灾难。

Security for AI vs AI for Security

Agent安全包含两个方向，企业最容易混淆和忽视的是前者：

| 方向 | 定义 | 紧迫性 | |------|------|--------| | Security for AI | 保护Agent系统本身不被攻击、滥用、越权 | 极高——Agent普及带来的新风险，企业严重缺失 | | AI for Security | 用AI技术提升安全运营效率 | 持续提升——AI赋能已有安全体系 |

关键区分：Security for AI是Agent安全的核心——因为Agent拥有执行能力后，自身就变成了攻击面和风险源。AI for Security则是将AI作为工具提升传统安全能力，不涉及Agent本身的安全治理。

五层解法体系

1. 沙箱隔离：让Agent在隔离环境运行，即使获得"最高权限"也无法破坏真实系统。Google Cloud Next 26推出的3box沙箱方案通过模拟内核技术实现，且自带暖池能力无额外延迟。

2. Agent身份与细粒度授权：像管理人类用户一样管理Agent身份。每个Agent应有独立身份，遵循最小权限原则，权限按任务动态分配、任务结束后回收。通过Agent Gateway管控所有通信，拦截提示注入攻击。

3. 人工审批关卡：对高风险操作设置人工确认节点。Google Cloud Agent Runtime支持断点保存恢复与零算力挂起——审批时释放算力，审批后快速恢复，不浪费资源。

4. 操作前预检 + 操作后审计：前置控制设定行为边界（类似高速公路护栏），后置审计全链路追踪每次工具调用和决策。Agent输出后做自我检测，不达标自动修订。

5. 分级权限策略：按风险等级设置不同控制策略——低风险自动执行事后审计，中风险记录日志异常告警，高风险需人工确认，极高风险双人审批不可自动执行。

OWASP Agentic AI 十大风险

OWASP已推出专门的Agent安全框架，核心风险包括：目标劫持、身份权限滥用、不安全的技能间通讯、提示注入、开源供应链风险、数据泄露、过度自主、幻觉导致危险行为、工具调用滥用、缺乏可观测性。

十大风险归结为两个核心维度：开放的权限（身份滥用、过度自主）和自主决策权（目标劫持、提示注入）。所有解法都围绕这两个维度展开。

与Agent Harness的边界

| 维度 | Agent Harness | Agent Security | |------|---------------|----------------| | 关注点 | 怎么让Agent跑得好、跑得稳 | 怎么让Agent跑得安全、跑得合规 | | 本质 | 工程化治理 | 风险合规 | | 典型手段 | 提示词约束、上下文管理、护栏 | 沙箱隔离、最小权限、身份治理、审计日志 | | 类比 | 操作系统（调度资源高效运行） | 安全合规部门（确保操作不越界） |

二者在"护栏"层面交集，但Security范围更广——还包括法规合规、供应链安全、身份治理等Harness不覆盖的领域。

法规硬约束

欧盟AI法案按四级风险分类管控：不可接受风险（禁止上线）、高风险（严格合规）、有限风险（透明度义务）、最小风险（无特殊要求）。高风险Agent需满足可审计、人类监督、数据治理等要求——这不仅是技术问题，更是法律合规问题。

例子

企业Agent安全架构：一家公司部署客服Agent，按五层体系设计——Agent运行在沙箱中（隔离），拥有独立身份只能访问客服数据库（最小权限），退款操作需人工审批（关卡），所有操作记录可追溯（审计），按风险分级控制（分级策略）。即使攻击者通过提示注入试图诱导Agent泄露数据，Agent既无法访问非客服数据（权限隔离），操作也被全程记录（审计可观测），高危操作被拦截（人工审批）。

相关概念

RELATED CONCEPT

Agent Harness

Agent Harness 是从模型调用到可治理 Agent 系统的工程化框架，用轻内核插件化架构、运行事实记录和数据库原生数据闭环让 Agent 可持续运行、可排查、可扩展。

RELATED CONCEPT

Loop Engineering

Loop Engineering 是围绕 AI Agent 设计可重复运行的反馈系统，让 Agent 能根据目标、上下文、工具结果、验证信号和停止条件持续迭代。

RELATED CONCEPT

Agent Orchestration

Agent Orchestration 是多模型/多Agent协作的编排能力——决定哪个模型做什么、何时做、如何组合，它是 AI 时代新的护城河。
