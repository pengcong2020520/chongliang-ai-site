DEFINED TERM

Agent Harness 是什么

Agent Harness 是从模型调用到可治理 Agent 系统的工程化框架，用轻内核插件化架构、运行事实记录和数据库原生数据闭环让 Agent 可持续运行、可排查、可扩展。

定义：Agent Harness 是模型外部的一整套驾驭系统，负责提示词管理与上下文构建、工具调用流程编排、运行状态与 Memory 管理、Skills/MCP/插件加载调度、运行轨迹记录与复盘、数据治理与评估反馈，使 Agent = Model + Harness。

适用场景

  • 生产级Agent系统的工具调用与状态管理
  • 多Agent协作的上下文隔离与交叉校验
  • Agent运行轨迹的调试与自我复盘
  • 企业级Agent从Demo到生产的数据治理演进

它不是什么

它不等于"调模型 API + 写 Prompt"——模型只是能力核心，Harness 决定模型在什么场景下使用什么工具、遵循什么边界、如何记录过程；也不等于传统 QA 或测试脚本——它关注 AI 系统的可复现、可观测和可迭代。

常见误区

  • 把"做Agent"等同于"调模型API+写Prompt"
  • 把所有能力内置到主流程中导致系统越来越重
  • 用普通聊天记录替代运行事实（Tap），无法支撑排查和复盘
  • 依赖文件系统管理Memory，不适合结构化查询和长期数据闭环

更完整的解释

很多人把"做 Agent"等同于"调模型 API + 写 Prompt"。但模型本身只是一个根据输入给出输出的函数，它不天然理解业务状态、历史轨迹和运行边界。真正让 Agent 在实际场景中稳定工作的，是模型外部的一整套驾驭系统——Harness。

轻内核 + 插件化架构

核心原则：维护一个稳定、清晰、轻量的内核，通过插件扩展业务能力。

内核只负责：Agent Loop 稳定运行、会话和状态管理、输入输出通道抽象、工具调用流程编排、上下文构建、运行轨迹记录。

插件负责扩展：MCP 工具能力、Memory 能力、IM 通道、定时任务、可视化界面、数据库适配、业务专属 Skills。

某个插件出问题时可以局部摘除或替换，不会直接破坏整个 Agent 系统。

阶段化拆分交互流程

Agent 的一次交互不是单一的"输入-输出"，而是多个明确阶段：接收输入 → 识别意图 → 构建上下文 → 选择工具 → 调用工具 → 记录运行事实 → 生成响应 → 写入运行轨迹 → 必要时复盘或沉默。

一个关键经验：Agent 不一定每次都要回复。 在多人协作或多 Agent 场景中，过多回复会变成噪音。输入通道和输出通道不应被强绑定。

Tap：运行事实而非普通聊天记录

传统聊天记录只保存"用户说了什么、Agent 回了什么"。但要支撑排查和复盘，需要记录的是运行事实（Tap）——Agent 识别了什么任务、使用了哪些工具、哪一步失败、做了什么判断、最终输出是什么。

Tap 的关键优势：既能给人看，也能给 Agent 自己看。 工程师和 Agent 可以基于同一份运行事实进行排查和复盘。Agent 可以读取自己的运行事实，对失败原因进行初步分析——这种设计将一部分工程排查能力前移到了 Agent 自身。

数据库原生 Harness

很多 Agent 项目依赖文件系统管理 Memory。这种方式对人类阅读友好，但不适合做结构化查询、统一治理和长期数据闭环。

数据库原生 Harness 的核心优势：所有运行数据统一存储、支持结构化查询、支持上下文/Memory/Trace/任务记录关联、便于权限管理和脱敏、便于沉淀评估数据集。

数据库不仅是业务数据存储组件，也是 Agent Harness 的运行底座。

数据闭环：从运行事实到自我复盘

当运行轨迹被结构化记录后，可以形成完整的数据闭环：对比不同模型在同类任务上的表现 → 评估同一模型的稳定性 → 沉淀失败案例 → 脱敏导出调试数据集 → 用于模型微调和评测。

Agent 系统评估维度

不能只看回答是否"看起来正确"：任务完成率、工具调用成功率、工具调用轮数、上下文利用质量、失败可解释性、扩展成本、数据闭环能力。

例子

一个生产级 Agent 系统的最小闭环：模型 API 可用 → 数据库可用 → 对话界面可访问 → Agent 能完成基础问答 → 运行记录可查询。在此基础上逐步添加 MCP、Skills 和业务插件。Agent 的运行轨迹不只是调试日志，而是用户信任系统的重要依据——能解释自己为什么这样回答、为什么失败、用了哪些工具，是面向真实业务落地的关键能力。用户可以直接问 Agent："刚才为什么没有回答成功？哪个工具调用失败了？"Agent 读取自己的 Tap 进行初步分析。

相关概念

RELATED CONCEPT

Harness Engineer

Harness Engineer 为 AI 系统建立可复现的运行、评测、日志、实验和反馈基座。

RELATED CONCEPT

Loop Engineering

Loop Engineering 是围绕 AI Agent 设计可重复运行的反馈系统，让 Agent 能根据目标、上下文、工具结果、验证信号和停止条件持续迭代。

RELATED CONCEPT

MCP (Model Context Protocol)

MCP 是 Anthropic 提出的模型上下文协议，为 AI 模型连接外部工具和数据源提供标准化接口，本质是一套治理协议，解决"谁有权做什么"。

RELATED CONCEPT

AI Skill

AI Skill 是可复用的 Agent 能力封装，内嵌完整 SOP 链路和所需工具，从手动传包正在演进为仓库化标准生态。

RELATED CONCEPT

Agent Orchestration

Agent Orchestration 是多模型/多Agent协作的编排能力——决定哪个模型做什么、何时做、如何组合，它是 AI 时代新的护城河。

RELATED CONCEPT

Context Engineering

Context Engineering 是设计和管理大模型在多轮交互中应该看到什么信息、何时看到、以什么形式看到的系统工程方法。

RELATED CONCEPT

Generator Paradox

发电机悖论揭示了 Agent 时代的生产力质变逻辑——单点替换不产生质变，系统级治理重构才产生质变。
