Spec-Driven Development 是什么

Spec-Driven Development 是用可执行规格替代传统 PRD 的协作范式——把业务需求转成研发和 AI 工具都能执行的结构化工程上下文。

定义：Spec-Driven Development 是一种将模糊业务需求约束为 AI Agent 和工程系统都能执行的结构化任务说明的方法，Spec 定义了输入输出、AI 能力边界、异常处理、验收标准和人工确认节点，使人、AI 和工程系统共享同一套可执行上下文。

适用场景

  • AI Coding 介入后的产品研发协作
  • Agent 系统的需求约束与能力边界定义
  • AI 项目的快速Demo与持续迭代
  • 需要AI能力边界、人工确认节点和验收标准的合规场景

它不是什么

它不等于"更长的 PRD"——Spec 的核心不是写得更长而是写得更可执行；也不等于替代研发拆需求——研发仍然要做技术方案设计和任务拆分，只是在一个更清晰的上下文上做工程判断。

常见误区

  • 认为 Spec 就是"更详细的 PRD"——Spec 是可执行的工程上下文，不是需求说明
  • 以为 Spec 可以替代研发拆需求——研发仍然需要做技术方案设计
  • 忽略 AI 能力边界定义——不定义边界，AI和研发会按"能自动就自动"方向做
  • 把 Spec 当一次性文档——Spec 需要随项目演进纳入版本管理

更完整的解释

AI 介入产品研发后，PRD 停在业务描述层的局限性被放大——AI Agent 最怕上下文模糊，而传统 PRD 恰恰把二次翻译留给了人。

PRD 的问题不是不好，而是停在了业务描述层

传统 PRD 主要回答三个问题：为什么做？给谁用？要做什么功能？对业务方来说够清楚，但研发拿到以后仍然要追问——录音支持什么格式？AI 生成结果能不能自动进入下一流程？哪些地方必须人工确认？验收标准是什么？

研发真正开始做工程时，还需要做一次二次翻译：把业务语言翻译成接口、数据、状态、任务、异常、测试和上线边界。这个二次翻译在传统开发中靠人的经验完成，但 AI Coding 介入后，这个环节成了最大的断裂点。

AI 就像一个能力很强但完全不了解业务的新同事。你跟他说"帮我做个需求分析功能"，他能做出来——但大概率不是你想要的那个。这不是 AI 笨，是你给它的上下文不够。

Spec 的核心：写得更可执行，不是写得更长

Spec 介于 PRD 和工程实现之间，它补充了传统 PRD 缺失的工程维度：

| 维度 | 传统 PRD | Spec 需要补充 | |------|---------|-------------| | 业务目标 | 说明为什么做 | 明确成功标准和不做什么 | | 用户流程 | 描述主流程 | 补状态、权限、异常和兜底 | | AI 能力 | 写"智能生成" | 明确输入、输出、置信度、人工确认 | | 验收 | 写功能可用 | 给测试样例和通过标准 |

AI 时代产品研发对齐的五个变化

1. 产品经理要定义 AI 的能力边界——AI 能做什么、不能做什么、哪些必须人工确认 2. 产品经理要提供样例，而不只是描述——输入样例、输出样例、Badcase、验收样例 3. 研发拆的不只是功能，还有 AI 链路——Prompt、RAG知识库、Agent工作流、置信度判断 4. 对齐方式从一次性交接变成持续校准——Spec初版 → 快速Demo → 发现Badcase → 补Spec → 再迭代 5. 验收从"功能可用"变成"结果可控"——回答是否有依据、是否幻觉、置信度低时是否兜底

产品经理的新角色：上下文设计师

产品经理不再只是需求的描述者，而要成为 AI 时代工程上下文的组织者——把业务、研发和 AI 之间的不确定性提前收敛。

Spec 从"人和人之间的对齐工具"变成了"人、AI 和工程系统之间的共享上下文"。

Spec 与 SDD 的关系

传统 SDD 强调"先写规格再写代码"——规格满足即终止。AI 时代，Spec 不只是代码的输入，它是 AI Agent 的运行约束：

  • Spec 定义了 AI 能做什么、不能做什么 → AI 能力边界
  • Spec 定义了输入输出格式 → Agent 调用的契约
  • Spec 定义了异常处理 → Agent 的兜底逻辑
  • Spec 定义了验收标准 → 验证器的判断依据
  • Spec 定义了人工确认节点 → Human-in-the-loop 的介入点

例子

需求："用户上传客户会议录音后，系统自动生成需求分析文档，并匹配历史项目。"

PRD 写到这里就够了。但 Spec 会进一步拆成：输入字段（audio_file、customer_id）、处理流程（ASR → 抽取需求 → 检索历史项目 → 人工确认）、AI 能力边界（不允许自动确认客户需求、不允许自动承诺交付周期）、异常处理（ASR失败提示重新上传、置信度低标记"需人工确认"）、验收标准（30分钟录音5分钟内生成初稿、必须包含来源引用、进入下一流程前必须有人确认）。研发不再从模糊 PRD 里猜业务边界，而是在清晰上下文上做工程判断。

相关概念

RELATED CONCEPT

Prompt Engineer

Prompt Engineer 通过任务建模、上下文组织、约束设计和评估迭代，让大模型在具体任务中更稳定地产出结果。

RELATED CONCEPT

Context Engineering

Context Engineering 是设计和管理大模型在多轮交互中应该看到什么信息、何时看到、以什么形式看到的系统工程方法。

RELATED CONCEPT

Loop Engineering

Loop Engineering 是围绕 AI Agent 设计可重复运行的反馈系统，让 Agent 能根据目标、上下文、工具结果、验证信号和停止条件持续迭代。

RELATED CONCEPT

Harness Engineer

Harness Engineer 为 AI 系统建立可复现的运行、评测、日志、实验和反馈基座。

RELATED CONCEPT

Vibe Coding

Vibe Coding 是自然语言驱动的编程范式——你描述感觉，AI 实现代码，编程从一门需要学习的技能变成了一个可以对话的能力。

RELATED CONCEPT

DSL

DSL 用领域语言压缩复杂性，让人、模型和系统之间共享更稳定的中间表达。
