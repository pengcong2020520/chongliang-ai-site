核心结论

企业AI PM（产品管理）平台的核心命题是：用AI重新定义需求→设计→开发的全链路流转，同时守住"关键决策必须人确认"的底线。 一套经过29个项目迭代验证的企业级AI PM平台，沉淀出固定交付SOP，验证了这条路径的工程可行性。平台分三大模块——任务工作台、Agent体系、Skill库与提示词库——并通过SDD（规格驱动开发）流程弥补国产大模型在Web Coding能力上的不足。本文从平台架构、开发适配、多Agent调度稳定性三个维度逐点拆解，并对照Anthropic Agent Skills的行业趋势做关联对比。

---

逐点拆解

1. 平台三大核心模块与流转规则

平台划分为任务工作台、Agent体系、Skill库与提示词库三个核心部分，形成"流转—执行—知识沉淀"的闭环。

任务工作台流转规则：

  • 需求录入后由AI自动分析，生成结构化拆解。
  • 核心关键点必须人工确认追责——这是平台设计的第一原则，AI可以提建议，但关键节点的决策权和责任归属在人。
  • 需求确认后，后续设计、开发环节的自动化程度很高，人工介入大幅减少。

Agent体系职责：

  • 结合历史项目知识库审查需求可行性，输出判断依据而非仅给结论。
  • 需求确认后接管开发编排，做任务拆解、定开发规则和技术栈，后续开发仅需实现功能模块。

Skill库与提示词库：

  • Skill将企业SOP沉淀为可发现、可激活、可执行的"技能组件"（与Anthropic Agent Skills理念一致）。
  • 提示词库管理各角色的系统提示词，确保Agent行为一致性和可追溯。

2. SDD规格驱动开发：弥补模型Coding短板

问题背景： 企业内部使用的是基于开源模型封装的大模型服务，Web Coding 能力弱于海外模型（如Claude、GPT系列），直接让模型生成完整代码质量不稳定。

适配方案——SDD（Specification-Driven Development）流程： 1. 开发前由Agent做好任务拆解：将需求拆成可独立实现的功能模块，定好每个模块的开发规则和技术栈。 2. 规格先行，代码后行：先输出明确的开发规格（接口定义、数据结构、业务逻辑规则），后续开发环节仅需按规格实现功能模块，降低对模型创造性的依赖。 3. 模块级验证：每个功能模块独立验证，避免"一错全错"的级联风险。

核心思路： 模型coding能力弱，就用"约束开发自由度"来补偿——把开放式代码生成降维成"按规格填空"，让模型的弱点不暴露。

3. 实战验证：29个项目迭代沉淀

实践背景： 一套企业级AI PM平台在企业内部独立搭建并跑通，推动公司AI化转型。

平台流程： 1. 业务方上传需求。 2. 通过RAG匹配知识库，判断需求可行性。 3. 输出标准化MRD、BRD、PRD文档。 4. 先开发MVP版本，与客户沟通调整。 5. 确认后交付开发团队做全自动化开发。

模块划分（与上文一致）： 任务工作台、Agent模块、Prompt与Skill库。

关键数据：

  • 已经过29个项目迭代，沉淀了固定交付SOP。
  • MVP先行策略降低了需求偏差风险，避免直接全量开发后返工。

4. 多Agent调度稳定性保障

核心问题： 多Agent并行工作时，容易出现调用混乱——Agent A调用了不该调用的Skill，用户越权触发高权限Agent等。

架构设计原则：

  • 区分用户和场景：在架构设计阶段就要明确不同用户角色对应的不同Agent集合。
  • Agent和Skill分层：Agent层负责编排和决策，Skill层负责执行，两层解耦。
  • 提前明确权限边界：

    • - 每个用户的Agent使用范围（哪些Agent可调）。 - 每个Agent的Skill调用范围（哪些Skill可执行）。

  • 避免调用混乱：通过白名单机制和分层设计，确保Agent只能在其被授权的Skill范围内操作。

这与OWASP提出的AI智能体安全风险中的"身份权限滥用"和"不安全的技能间通讯"风险直接对应。 分层+白名单是工程上最可控的防御手段。

关联与对比

与Anthropic Agent Skills的对照

| 维度 | 企业级AI PM平台实践 | Anthropic Agent Skills (2025-10) | |------|-------------------|----------------------------------| | 核心理念 | Skill库+提示词库沉淀企业SOP | "Don't Build Agents, Build Skills Instead" | | Skill定位 | 可执行的企业SOP组件 | 可发现、可激活、可执行的技能组件 | | 架构特征 | Agent和Skill分层，权限白名单 | 渐进式披露（Progressive Disclosure），降低60%-80% Token消耗 | | 适用场景 | 企业内部PM全流程 | 通用Agent生态 |

关键发现： 实战验证与Anthropic的Agent Skills理念高度一致——都把Skill作为独立于Agent的可复用组件，而非把能力硬编码在Agent内部。但企业级实践更强调权限分层和追责机制，这反映了企业内部场景对合规和安全更高要求。

与Claude Code市场表现的关联

Claude Code 等工具的快速增长验证了"AI辅助开发"的市场需求。SDD流程是在模型能力受限场景下的工程适配方案——当可用模型的 Coding 能力不足以支撑自主开发时，用规格约束替代模型自主性，让能力较弱的模型也能完成任务。SDD 的约束程度可根据模型能力动态调整：模型越强，约束越少；模型越弱，约束越严。

企业实践路径的行业共性

行业实践路径是从"VOC分析""内容生成助手"等低风险场景切入，逐步扩展到核心业务（如货品全生命周期管理），为10000+门店服务。这套AI PM平台的路径高度类似：从需求分析切入，逐步扩展到全链路自动开发，29个项目迭代本身就是"渐进式扩展"的体现。

与AI安全风险框架的关联

多Agent调度的权限分层设计，直接回应了OWASP AI智能体十大风险中的：

  • 目标劫持：通过Agent使用范围白名单限制。
  • 身份权限滥用：通过用户-Agent-Skill三层权限映射。
  • 不安全的技能间通讯：通过Agent和Skill分层解耦，隔离通讯通道。
