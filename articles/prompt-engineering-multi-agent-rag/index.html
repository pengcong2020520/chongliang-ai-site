核心结论

Prompt工程、多Agent架构、RAG（检索增强生成）是当前Agent工程化的三类基础设施技术。行业公开调研显示，这三类技术已成为企业级AI应用的标配组件。三者并非互斥关系，而是在不同环节各自发挥作用的互补技术栈：Prompt工程解决"如何与LLM高效沟通"，多Agent架构解决"如何将复杂任务拆分与分派"，RAG解决"如何让LLM基于私有知识准确回答"。

本文结合公开研究资料与工程实践，对三类技术的核心原理、适用场景与常见误区进行系统梳理。

---

逐点拆解

一、Prompt工程

核心原理

Prompt工程的核心在于利用LLM对输入文本的注意力分布特性，设计能最大化引导正确输出的提示词结构。认知科学中的首因效应和近因效应在LLM中同样存在体现：

  • 首尾效应（primacy/recency effect）：LLM对prompt开头和结尾的内容赋予更高权重。关键指令放在首尾位置时，模型遵从度显著提升。
  • 强制性用语权重提升：使用"必须""不得""务必"等强制性措辞时，模型对该约束的关注度提高，执行率上升。
  • 少样本案例的作用：在prompt中提供少量输入-输出示例（few-shot），能有效校准模型的输出格式与风格，是降低hallucination的重要手段。

适用场景

  • 快速验证LLM在特定任务上的能力边界
  • 对输出格式有严格约束的结构化生成任务
  • 不具备RAG基础设施时的轻量级知识引导

常见误区

⚠️ 忽略少样本案例的作用——仅依赖零样本指令，导致输出格式不稳定或偏离预期。实践中应优先考虑在prompt中嵌入2-3个高质量示例。

---

二、多Agent架构

核心原理

多Agent架构采用"意图识别 + 专家分发"的模式运作：

1. 意图识别层：接收用户输入，判断应分发给哪个专家Agent处理 2. 专家分发层：将请求路由至对应的领域专家Agent 3. 主流分层架构：各专家Agent独立部署、独立迭代，互不影响

这种架构的核心优势在于解耦——每个Agent可以独立优化、独立升级、独立监控，避免了单体prompt在复杂场景下的维护困难。

适用场景

  • 业务领域多样、单一Agent难以覆盖全部意图的场景
  • 不同专家需要接入不同工具/知识库的场景
  • 团队需要并行开发多个Agent能力的组织架构

常见误区

⚠️ 过度拆分Agent——意图分类过细会导致分发准确率下降，且增加系统复杂度。应在"专家粒度"和"分发准确率"之间取得平衡。

---

三、RAG（检索增强生成）

核心原理

RAG遵循线下处理 → 线上匹配的标准流程：

线下处理流程：

| 步骤 | 说明 | |------|------| | 文档解析 | 将PDF、PPT等非结构化文档解析为可处理格式 | | OCR | 通过OCR工具转换为结构化文本数据 | | 数据清洗 | 去除噪声、修正格式异常 | | 切块 | 调整chunk大小，平衡检索精度与上下文完整性 | | 向量化 | 将文本块编码为向量表示 | | 入库 | 写入向量数据库，建立索引 |

线上匹配流程：

| 步骤 | 说明 | |------|------| | Query向量化 | 将用户查询编码为向量 | | 相似度检索 | 在向量库中检索最相关的文档块 | | Context注入 | 将检索结果作为上下文拼入prompt | | LLM生成 | LLM基于注入的上下文生成最终回答 |

业务驱动力

在需求分析与可行性评估场景中，必须匹配历史沉淀的项目文档。这一业务需求直接驱动了RAG系统的引入——仅靠LLM的参数化知识无法覆盖企业私有文档，必须通过检索机制将历史项目文档注入生成过程。

适用场景

  • 需要基于企业私有知识库回答问题的场景
  • 知识频繁更新、不适合重新训练模型的场景
  • 需要可追溯来源的合规性场景

常见误区

⚠️ 忽视chunk大小调优——chunk过大导致检索精度下降（检索到大量无关内容），chunk过小导致上下文不完整（丢失关键信息）。需根据文档类型和业务场景反复实验调优。

四、工具调用方案对比（补充）

在Agent工程化实践中，工具调用是连接LLM与外部系统的关键环节，主流方案有两种：

| 维度 | CLI工具 | MCP工具 | |------|--------|---------| | 调用机制 | 依赖大模型内置的bash原生工具调用 | 标准化协议，独立于模型 | | 权限管控 | 弱，难以细粒度控制 | 强，满足大型企业鉴权需求 | | 适用场景 | 快速原型、个人开发 | 登录/注册/数据库调用等企业级场景 | | 安全性 | 较低 | 较高，支持权限隔离 |

登录、注册、数据库调用等涉及鉴权与权限管控的场景必须使用MCP方案。

关联与对比

三类技术的定位关系

用户请求
  │
  ├─→ 多Agent架构（意图识别 + 专家分发）
  │       │
  │       ├─→ 专家Agent A
  │       │     ├─→ RAG（检索私有知识）
  │       │     └─→ Prompt工程（引导输出格式）
  │       │
  │       └─→ 专家Agent B
  │             ├─→ 工具调用（CLI / MCP）
  │             └─→ Prompt工程（引导输出格式）

三者并非平级竞争关系，而是嵌套互补关系：

  • 多Agent架构是系统级的组织方式，决定"谁来做"
  • RAG是知识级的增强手段，决定"用什么知识做"
  • Prompt工程是交互级的优化手段，决定"怎么跟LLM说"

核心适用场景速查

| 技术 | 解决的核心问题 | 典型场景 | |------|---------------|---------| | Prompt工程 | LLM输出不可控 | 格式约束、能力验证、轻量引导 | | 多Agent架构 | 单Agent能力边界与维护成本 | 多领域意图分发、并行开发 | | RAG | LLM缺乏私有知识 | 企业文档问答、需求分析、合规溯源 |

实践检验要点

判断对三类技术的理解是否到位，三条标准： 1. 能否清晰说明三类技术各自的核心适用场景（而非混淆边界） 2. 能否指出常见误区（体现实践经验而非纸上谈兵） 3. 能否说明三类技术如何协同工作（体现系统设计思维）
