核心结论

当讨论"用Agent操作广告投放"时，大多数人的第一反应是做一个能听懂自然语言的机器人，对接广告平台API，投手说一句就执行一步。

这个理解只触及了表层。广告投放Agent的工程本质，是一套分层调度系统：大模型理解意图并拆解策略，中型模型调度API执行操作，小模型监控数据并生成反馈，多模态模型辅助创意生产。每一层承担不同的职责，使用不同规格的模型，追求不同的性能目标。

这篇文章聚焦的不是"投手角色怎么变"（那是另一篇文章的主题），而是技术架构怎么设计——平台生态能力如何映射为Agent工具链，Function Calling与MCP协议如何选择与迁移，安全治理框架怎么建，实施路径分几步走。

二、广告平台生态能力矩阵：Agent化的基础

2.1 Marketing API能力全景

主流广告平台（腾讯广告、Google Ads、Meta Ads等）的Marketing API已经形成了高度标准化的能力矩阵。Agent化的第一步，是将这些API能力映射为可调用的工具：

| 能力域 | API能力 | Agent化方向 | |--------|---------|------------| | 账户管理 | 余额查询、充值记录 | 小模型定时监控余额，低于阈值自动告警 | | 投放管理 | 计划/组/创意的CRUD | 大模型理解意图→中型模型调度API创建 | | 定向能力 | 人口属性/兴趣/行为/lookalike | 结合受众画像自动生成定向包 | | 数据报表 | 消耗/展示/点击/转化实时数据 | 小模型生成日报/周报，异常自动归因 | | 创意管理 | 素材上传/审核状态/智能帧 | 多模态模型生成文案+封面，自动A/B测试 |

这张表的关键洞察是：不同能力域适合不同规格的模型。账户监控和数据汇总用小模型即可（低延迟、低成本）；投放管理需要意图理解和策略拆解，用大模型；创意生成涉及多模态，用专用模型。这就是分层架构的现实驱动力——不是因为"多模型"听起来更先进，而是因为不同任务的计算特征完全不同。

2.2 平台原生智能化能力

广告平台自身也在快速AI化。以腾讯广告为例：

智能帧能力：系统自动识别视频内容，优选画面清晰、内容丰富的帧作为封面。对短剧类广告投放尤其关键——短剧的画面质量参差不齐，人工选帧耗时且主观，AI选帧可以基于历史CTR数据持续优化封面选择策略。

元器平台与MCP协议：腾讯元器平台已支持MCP协议接入，可构建企业级智能体并发布至企业微信及小程序。Google Ads已推出独立MCP Server。这意味着广告平台的API正在从"开发者对接"模式向"Agent原生"模式演进——平台自己开始提供标准化的Agent工具接口，而非仅提供REST API让开发者自行封装。

这个趋势的深层含义是：Agent工具链的构建成本会持续下降。 今天需要自定义封装的接口，明天可能平台直接以MCP工具形式提供。所以工具链架构必须设计成可热插拔的，不能把工具绑定死在某个特定的封装方式上。

三、分层调度架构：四层模型协作

3.1 架构全貌

一套完整的广告投放Agent系统，建议采用四层架构：

┌─────────────────────────────────────────────┐
│           用户交互层（大模型）                │
│  理解"帮我给XX产品投信息流广告，目标ROI=1.5"  │
│  → 拆解为：选品分析/定向策略/出价方案/创意规划  │
├─────────────────────────────────────────────┤
│           工具调度层（中型模型）               │
│  调用广告平台API创建计划 → 设置定向 → 上传创意  │
│  Function Calling + MCP协议                  │
├─────────────────────────────────────────────┤
│           监测反馈层（小模型）                 │
│  实时消耗/ROI监控 → 异常告警 → 日报/周报生成    │
│  低延迟流式处理                               │
├─────────────────────────────────────────────┤
│           创意生成层（多模态模型）              │
│  广告文案生成、素材A/B测试变体                 │
└─────────────────────────────────────────────┘

每一层的设计逻辑：

  • 用户交互层用大模型（pro级），因为意图理解和策略拆解需要强推理能力。用户说"目标ROI=1.5"，模型要能拆解为选品分析、定向策略、出价方案、创意规划四个子任务，并为每个子任务生成工具调用参数。
  • 工具调度层用中型模型，执行Function Calling或MCP协议调用。这层不需要强推理，但需要高精度地生成API参数——一个出价参数错了，可能就是真金白银的损失。
  • 监测反馈层用小模型（mini级），因为监控和报告是高频低复杂度任务，需要低延迟。实时消耗监控、异常告警、日报生成，这些都是"快速但不需要太聪明"的任务。
  • 创意生成层用多模态模型，生成广告文案和素材变体。这层独立出来是因为多模态模型的调用成本和延迟都高，不适合放在主链路中。

3.2 双模型架构的工程权衡

在实际项目中，最核心的权衡是：要不要分层？分几层？

最简单的方案是单一大模型——一个pro级模型包揽所有工作。问题是推理太慢：复杂需求需要90秒才能返回结果。这对投放场景是不可接受的——投手在钉钉里发一条指令，等一分半钟才出结果，体验极差。

双模型架构（大模型+小模型）把90秒压缩到20-30秒。大模型负责"想清楚"（意图理解+策略拆解），小模型负责"做快速"（结果汇总+日报生成）。这不是理论推演，而是工程实测的结论。

但分层也有代价：层间通信增加系统复杂度。 大模型拆解出的子任务，需要准确地传递给中型模型执行；中型模型的执行结果，需要小模型汇总。任何一层的信息丢失都会导致整体错误。所以分层不是越多越好——四层是实践中比较平衡的设计，再多就需要极强的工程治理能力。

3.3 工具链分类：21个工具的三类划分

一个广告投放Agent的工具链，典型规模在20个左右，分三类：

| 类型 | 说明 | 示例 | |------|------|------| | 内部自定义工具 | 匹配业务数据的查询接口 | 产品对应账户查询、投放历史记录查询 | | 官方MCP/API工具 | 广告平台官方提供的标准化接口 | 修改出价、创意操作、报表拉取 | | 自定义封装接口 | 处理官方工具未覆盖的操作 | 跨平台数据聚合、业务规则封装 |

工具设计的核心原则是：原子能力固定，组合方式由大脑决定。 工具本身是"哑的"——每个工具只做一件事，不包含业务逻辑。业务逻辑由大模型在调用时决定：先调哪个工具、后调哪个工具、用什么参数、怎么组合结果。

举例：用户说"查一下今天消耗超过500的广告"，Agent自动拆成两步——第一步调用工具查询对应所有账户，第二步调用工具筛选符合消耗条件的广告，最后由小模型整理结果输出。用户不需要知道背后调了哪几个API，只需要表达意图。

这个设计的关键好处是可扩展性：新增一个广告平台、新增一个业务规则，只需要增加对应的工具，不需要改Agent的核心逻辑。大模型自动发现新工具并决定何时调用。

四、Function Calling vs MCP：工具协议的选择与迁移

4.1 两种协议的对比

| 维度 | Function Calling | MCP协议 | |------|-----------------|---------| | 适用场景 | 单一Agent内部工具调用 | 多Agent跨系统工具共享 | | 集成成本 | 低（模型原生支持） | 中（需部署MCP Server） | | 工具发现 | 预定义，静态 | 动态发现，可热插拔 | | 生态成熟度 | 成熟，主流模型均支持 | 快速增长中，平台级支持出现 |

4.2 迁移路径：先Function Calling，后MCP

MVP阶段优先使用Function Calling。 原因很简单：集成成本最低，模型原生支持，不需要额外部署基础设施。在这个阶段，目标是验证核心链路——意图理解→工具调用→结果返回——是否跑通。协议选择不是瓶颈，场景验证才是。

规模化后迁移到MCP。 当Agent从单一边界扩展到多Agent协作（投放Agent、创意Agent、监控Agent分离），或者需要对接多个广告平台时，Function Calling的静态工具定义就成了瓶颈。MCP协议的动态发现和热插拔能力，让新工具可以在不停机的情况下加入系统。

迁移的关键设计是工具抽象层：在Function Calling和实际API调用之间加一层抽象，让工具定义与协议无关。这样迁移到MCP时，只需要替换工具注册和发现机制，不需要重写工具逻辑。

4.3 平台MCP化的趋势判断

Google Ads已推出独立MCP Server，腾讯元器平台已支持MCP协议接入。这意味着广告平台正在从"提供API让开发者自己封装"转向"直接提供Agent原生工具"。

对Agent架构的影响是：工具链的维护成本会持续下降。 今天需要自定义封装的接口，未来平台可能直接以MCP工具形式提供。但短期（1-2年）内，自定义封装仍然是必要的——平台MCP工具的覆盖度还远不够，很多业务特定的操作（如跨平台数据聚合）平台不会提供。

五、IM+Agent交互架构：入口设计的三种范式

5.1 架构模式

广告投放Agent的交互入口，主流选择是嵌入IM机器人：

广告主（自然语言指令）
    │
    ▼
IM机器人（飞书/钉钉/企业微信）
    │
    ▼
投放调度Agent（大模型理解意图）
    ├──→ 广告平台API（创建/调整投放计划）
    ├──→ 创意生成Agent（多模态模型生成素材）
    └── 数据监控Agent（小模型实时监控）
    │
    ▼
IM消息推送（结果反馈）
    - 投放创建确认
    - 消耗/ROI日报
    - 异常告警（如消耗过快/ROI下降）

这个设计的核心价值不在"做了一个功能强大的机器人"，而在于设计了最短的操作路径。Agent落地最大的障碍不是技术，而是采纳——要求用户学一个新工具的摩擦成本极高。把Agent嵌入IM，等于让AI出现在用户已经在用的地方，零学习成本即可上手。这不是技术决策，是采纳策略决策。

5.2 平台选择

| 平台 | 特点 | 适用场景 | |------|------|---------| | 飞书 | 多维表格+Agent构建低代码自动化工作流 | 灵活配置的投放团队 | | 钉钉 | AI智能体平台+消息Channel | 强流程管控的大型团队 | | 企业微信 | 与腾讯生态原生打通 | 直接对接腾讯广告API |

选择的关键不是哪个平台"更好"，而是团队已经在用哪个。Agent的价值在于嵌入已有工作流，而不是制造一个新的工作流。如果团队用钉钉，就做钉钉机器人；用飞书，就做飞书机器人。不要为了技术先进性强制迁移团队的工作平台。

六、安全治理框架：广告投放场景的特殊性

广告投放Agent的安全治理比一般Agent更关键，因为每一步操作都直接关联资金。

6.1 操作分级管控

| 操作级别 | 示例 | 管控措施 | |---------|------|---------| | 只读操作 | 查询消耗、查看报表 | 无需确认，直接执行 | | 常规写操作 | 修改出价、调整定向 | 记录日志，可回溯 | | 敏感操作 | 充值、大额预算修改 | 需人工确认后执行 | | 关键操作 | 创建投放计划、修改预算 | 二次确认机制 |

分级的逻辑是：操作的不可逆性和资金影响程度决定管控级别。查询操作无副作用，直接执行；修改出价可回滚，记录日志即可；但充值和大额预算修改一旦执行很难撤回，必须人工确认。

6.2 密钥与权限管理

  • API密钥通过环境变量或密钥管理服务注入，不暴露给模型。模型只知道"调用这个工具"，不知道工具背后用什么密钥认证。
  • 操作日志全量记录，支持审计回溯。每一条工具调用都记录：谁调的、调了什么、什么参数、什么时间、返回什么结果。
  • 权限按角色隔离——不同用户能调用的工具范围不同。比如普通投手不能直接充值，只有管理员可以。

6.3 记忆与状态管理

广告投放Agent需要管理四类记忆：

| 记忆类型 | 内容 | 用途 | |---------|------|------| | 投放历史记忆 | 历史投放策略及效果 | 避免重复试错，辅助策略推荐 | | 用户偏好记忆 | 投放习惯、预算偏好、风险偏好 | 个性化默认参数，减少交互轮次 | | 对话上下文 | 多轮对话中的策略调整上下文 | "上次提到的广告"自动补全 | | 异常事件记忆 | 历史异常及处理方式 | 未来类似异常自动处理 |

记忆管理的核心矛盾是：记住太多会干扰决策，记住太少会重复交互。 实践中的平衡点是：对话上下文默认保留最近3-5轮，用户偏好持久化存储但可手动清除，异常事件记忆按"同类异常出现3次以上"的阈值决定是否纳入自动处理库。

七、实施路径：四阶段渐进式落地

| 阶段 | 目标 | 关键动作 | 验证指标 | |------|------|---------|---------| | Phase 1（MVP） | 跑通核心链路 | IM机器人+单一大模型+广告平台API Function Calling | 意图识别准确率、基础操作成功率 | | Phase 2（优化） | 双模型+告警 | 引入双模型架构，添加异常告警和日报自动生成 | 响应延迟、告警准确率 | | Phase 3（规模化） | 多Agent协作 | 迁移MCP协议，投放/创意/监测分离，对接更多广告平台 | 工具复用率、跨平台覆盖度 | | Phase 4（智能化） | 自动优化 | 强化学习自动优化出价，基于历史数据自动生成投放方案 | ROI提升幅度、人工干预率 |

Phase 1的核心验证

Phase 1不是在做产品，是在做假设验证。要验证的核心假设是：投手愿意用自然语言下指令，且Agent能准确理解并执行。这个阶段用最简单的技术栈——单一模型、Function Calling、最少的工具（5-8个核心工具即可）——快速跑通"说一句话→执行操作→返回结果"的闭环。

如果Phase 1验证失败（投手不愿意用，或者意图识别准确率不够），不要急于加功能，而是回头检查场景选择是否正确。40%的Agent项目失败率（Gartner数据）的核心原因不是技术不够强，而是场景选择错误。

Phase 2的架构升级

Phase 2引入双模型架构是第一个大的工程升级。关键变化是：大模型不再直接返回结果给用户，而是把工具调用的原始结果交给小模型做汇总和格式化。这个变化看似简单，但涉及整个请求链路的重构——从同步返回变成异步处理，需要设计任务队列和结果回调机制。

同时引入的异常告警和日报自动生成，是小模型价值的第一次体现。这些任务不需要强推理，但需要高频执行和低延迟——正是小模型的擅长领域。

Phase 3的协议迁移

Phase 3从Function Calling迁移到MCP，是架构层面的"换地基"。前面提到的工具抽象层在这个阶段发挥作用——如果Phase 1设计了良好的抽象层，迁移成本可控；如果没有，这一步几乎等于重写工具链。

多Agent协作（投放Agent、创意Agent、监控Agent分离）也在此阶段引入。分离的驱动力不是"多Agent更先进"，而是不同的Agent有不同的SLA要求——监控Agent需要7×24低延迟运行，创意Agent可以批量异步执行，投放Agent需要高精度实时响应。混在一起部署，SLA互相拖累。

Phase 4的智能化跃迁

Phase 4引入强化学习做出价优化和投放方案自动生成，是从"执行Agent"到"决策Agent"的跃迁。这个阶段的技术栈与前三个阶段有本质区别——需要训练数据积累、模型微调管道、A/B测试框架。绝大多数项目到不了Phase 4，也不需要到——Phase 2-3已经能覆盖80%以上的业务价值。

八、关键设计判断

1. 分层架构不是炫技，是任务特征的必然要求。 不同能力域的计算特征不同（监控要快、创意要多模态、调度要精确），用一个大模型包揽所有任务，在性能和成本上都不优。

2. 工具链的可插拔性比工具数量更重要。 21个工具不是重点，重点是新增一个工具不需要改Agent核心逻辑。平台API在持续演进，工具链必须能跟上。

3. 安全治理要在Phase 1就设计好，不是Phase 3再补。 广告投放涉及真金白银，等出事了再加安全机制就晚了。操作分级、密钥隔离、日志审计，这三件事从第一天就要有。

4. IM入口的选择是采纳策略决策，不是技术决策。 最好的Agent入口是用户已经在用的那个IM，而不是技术上最先进的那个。

5. Function Calling到MCP的迁移是渐进的，不是一刀切。 先用Function Calling验证场景，规模化后再迁移。中间靠工具抽象层降低迁移成本。

6. 四阶段实施路径的核心逻辑是：每一步都在验证一个假设。 Phase 1验证"用户愿不愿意用"，Phase 2验证"双模型架构能不能提速"，Phase 3验证"多Agent协作能不能扩展"，Phase 4验证"AI能不能自主决策"。任何一步验证失败，不要硬推下一步。

"大模型其实没有任何能力，就是指挥干活，它就指挥你本地的工具去执行。优秀的agent设计：能CPU处理的我CPU处理，实在不行的我让大模型去判断。原子能力都是固定的，但是大模型作为大脑决定怎么组合工具，满足不同的个性化需求。"

这句话道出了广告投放Agent工程化的本质：模型的智能体现在工具组合的方式上，而不是工具本身。 工具是固定的原子能力，架构是固定的分层调度，唯一灵活的是大模型如何理解意图、拆解策略、组合工具。把这三件事做对，Agent就能从"能用的机器人"变成"好用的投放助手"。
