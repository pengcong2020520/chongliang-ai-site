自研Agent中枢系统架构：多工具管控、上下文桥接与B端商业化判断

核心结论

问题背景

企业政企项目原有交付链路极长：客户经理拿到需求后，需要与客户、供应商、内部售前反复沟通确认，仅沟通环节就耗时一个月以上，从需求到交付至少半年。引入AI后，单人可搭建从需求录音→RAG匹配→需求拆解→输出MRD/BRD/PRD→web coding生成demo→客户确认→工程化交付的完整工作流，交付周期大幅压缩，累计完成数十个项目。

但工作流只解决"一个人怎么用AI"的问题。当使用者扩展到几百人——客户经理、产品、开发团队——每人可能同时使用多个AI工具（如Claude Code、Codex、Dify等），就需要一套中枢系统做统一管理。

与开源平台的关键差异

Dify类开源平台的核心局限在于：

  • 仅支持简单的文档上传，无法满足企业多品类文档（源代码、招投标文件、PRD、技术方案）的沉淀与复用需求
  • 不支持多Agent统一管理，无法在同一项目下实现多个AI工具的无缝切换
  • 缺乏工具集成层，没有统一的Tool Hub做底层MCP/CLI工具的集成与调度

自研中枢系统的定位是：不只做工作流，还要做RAG知识库自迭代、Agent/Skill管理、多工具统一调度——是一个可复刻到其他同类企业的通用型平台。

二、Agent中枢系统核心架构

系统分四大核心模块：

模块一：任务工作流

拿到客户需求会议录音后，分两条链路处理：

1. RAG匹配链路：将需求与历史项目知识库做匹配。匹配到现成项目的，仅需AI做简单改造即可交付 2. 新需求链路：未匹配到现成项目时，通过AI做需求拆解，输出MRD、BRD、PRD文档，走内部评审后快速开发demo给客户确认，确认后交给工程化团队做正式交付

模块二：RAG知识库（自迭代）

  • 内置企业资产知识库，新项目交付产生的新文档自动回滚沉淀到知识库
  • 不同项目、不同用户、不同场景的RAG知识库做隔离处理
  • 数据迭代采用闭环方式，通过哈希值判断知识点更新，自动替换原有知识

模块三：Agent与Skill工具模块

  • 内置适配交付场景的各类Skill与Agent：会议助手、文档输出、PPT生成等
  • Skill按业务SOP封装，包含触发条件、完整流程链路、可调用工具、bad case避坑限制、最佳实践示例
  • 可服务全公司使用

模块四：多工具管控（核心差异点）

这是系统区别于普通工作流平台的关键模块：

用户(SSO身份识别)
  ├── Agent注册中心(每人绑定多个Agent)
  │     ├── Claude Code
  │     ├── Codex
  │     ├── Dify
  │     └── ...其他AI工具
  ├── API Key统管(统一管理多工具密钥)
  ├── 上下文桥接(本地客户端进程 ↔ 服务端数据库同步)
  └── Tool Hub(底层MCP/CLI工具集成)

SSO身份识别：用户进入系统后通过企业SSO完成身份认证，确定"谁在用什么"。

Agent注册中心：每个用户可能使用多个AI工具，系统为每个用户建立Agent绑定关系，支持一人多Agent的管理模式。

API Key统管：统一管理用户使用的多个AI工具的API Key。在央企等受控环境下，国外模型的API使用走线下报销流程，由用户在本地客户端使用工具，系统仅做监控与管理。

上下文桥接：这是技术上最有特色的部分。系统包含一个本地客户端，在用户电脑上运行一个进程，与服务端数据库做同步。当用户在不同AI工具之间切换时（如从Claude Code切到Codex），通过这个同步机制实现上下文的无缝桥接——用户不需要在每个工具里重新描述项目背景。

Tool Hub：底层集成MCP协议工具和CLI工具，通过统一调度平台实现工具的注册、发现和调用。这是实现"同一项目下多AI工具无缝切换"的基础设施。

三、企业环境下的AI工具适配策略

合规约束下的模型选型

央企环境对国外模型管控严格，不能使用公有云大模型。适配方案：

  • 基础大模型：本地部署DeepSeek，无额外基础费用
  • 开发工具：开发团队线下使用Claude Code等工具，token消耗通过项目金额覆盖
  • 成本监控：所有工具的token消耗统一在平台中监测管理

推广策略与阻力

平台要求相关业务线全员使用（数百人规模），推广策略包括：

  • 设定考核指标，强制要求各部门使用
  • 内部使用频次通报，领导知晓
  • 专项培训支持

但推广仍面临阻力：员工改变意愿低、学习成本高、原有工作习惯难以打破。核心认知：如果AI工具不能切实提升效率，用户会将其视为额外负担而非提效工具。AI替换企业原有节点只能先提升单点效率，需要所有节点都替换完成后才会出现整体效率的爆发式提升。

四、B端AI产品商业化方向判断

单点切入 > 全链路方案

"在B端市场里面，解决一个点的需求，市场会比解决全链路更大一点。"

核心逻辑：将全流程打散为模块化售卖，聚焦单个亮点需求切入B端市场。典型案例——某公司仅靠解决"多文档喂给AI"这一单点需求，就获得了大量收益和强售卖竞争力。全链路解决方案看似完整，但市场教育成本高、决策链路长，反不如单点突破。

Web Coding将成为基础办公技能

判断未来web coding不会是AI从业者的专属技能，会像WPS一样成为所有办公人员的基础技能。每个人都可以搭建适配自身岗位的Agent和Skill，通过"一人+AI"的模式扩大现有生产力。

企业级Agent OS的机会

当企业内部全员开始搭建个人Agent后，会产生以下管控需求：

  • 工具注册：统一管理企业内部所有AI工具的注册与发现
  • Token管控：监控和管控全公司AI工具的token消耗与成本
  • 多Agent上下文管理：管理不同Agent之间的上下文传递与权限
  • Agent权限治理：管控Agent的访问权限、数据权限、操作权限

这构成了"企业级Agent操作系统"的产品机会——不是做某个具体的AI工具，而是做管理所有AI工具的中枢层。

AI工程化演进四阶段

行业AI工程化演进路径： 1. Prompt工程：通过提示词优化LLM输出 2. 上下文工程：管理系统上下文，提升Agent连贯性 3. Harness工程：管理大模型以外的所有能力（记忆模块、工具管理、权限管理） 4. Loop工程：提示词形成自我闭环、自动化迭代

Agent = Model + Harness。Harness是模型以外的一切——记忆、工具、权限、编排逻辑。Agent中枢系统本质上就是在做Harness工程。

五、Skill执行框架选择

React vs Plan-and-Execution

| 维度 | React框架 | Plan-and-Execution框架 | |------|-----------|----------------------| | 自主性 | 强，实时推理决策 | 弱，按预设计划执行 | | 适用场景 | 输出前后依赖性强、需实时生成 | 目标收敛、主干链路清晰、固定格式要求 | | 企业适配 | 适合探索性、创造性任务 | 适合企业内部固定流程场景 | | 风险 | 自主性过强可能脱离约束 | 灵活性不足 |

企业实践判断：企业内部多数固定流程场景适合Plan-and-Execution框架，因为企业内部每个节点的文档都有固定格式要求，React的自主性太强容易导致输出脱离约束。信息收集、需求方向梳理等链路确定的场景，采用先规划再执行的方案效果更好。

Skill新建判断标准

  • 主干流程SOP变更 → 新建Skill
  • 主干流程不变、仅某节点调整 → 迭代现有Skill
  • 主干流程完全不同 → 新建Skill

核心原则：Skill内部有一套SOP骨架（第一步做什么、第二步做什么），骨架变了就是新Skill，骨架没变只是节点微调就迭代现有Skill。

六、RAG全链路追踪调优

问题

传统RAG只知道整体准确率不高，无法定位是哪个环节出了问题。

方案：Run ID全链路追踪

引入Run ID串联整条RAG链路，在切块、文档解析、embedding、检索、重排序等每个环节都通过Run ID记录数据，建立约8张数据表。当问答准确率不足时，通过Run ID查询每个环节的日志数据，快速定位具体问题模块。

调优实践

文档解析调优：

  • 印刷体使用DeepSeek OCR识别
  • 手写体更换为Pandora OCR识别
  • 跨页表格增加前后首尾检查，通过表头迁移、文本拼接处理内容中断

文本切块三层调优： 1. 第一层：按段落切分 2. 第二层：按语句二次切分 3. 第三层：语义分析调整——语义关联度高的切块重新拼接，语义差异大的内容再次切分

输出质量验证：

  • 强制要求大模型输出必须带上内容来源索引
  • 引入AI自动判别引用来源和输出内容的关联性
  • 上线前通过预置答案的评测集做自动匹配测试
  • 上线后邀请小批量用户体验收集bad case，回流到全链路做调优

调优后整体文档处理准确率从80%出头提升到95%以上。

七、知识管理与迭代机制

知识收集：管理手段 > 技术手段

企业知识分散在各部门，无法靠技术自动收集。核心方法： 1. 根据项目覆盖场景完成调研，梳理需要的知识清单 2. 明确对应知识所属部门 3. 下发给对应部门按清单上报 4. 每个部门安排固定对接人，按每周周期定期上报更新 5. 对未按时上报的部门发内部通报推动

关键认知：知识收集本质是管理问题，不是技术问题。 技术层面通过哈希值判断识别知识点的更新，自动替换原有知识。但对于部门隐瞒知识更新未上报的情况，需要设计case驱动的迭代策略——当错误率突然升高或出现接不住的问题时，产生变动信号，流转给知识供给方协同解决。

模糊问题处理策略

开放问答场景中用户表达模糊时的处理策略：

  • 默认策略：大模型假设补全，直接给出回答（避免多轮澄清拉长链路引发用户不耐烦）
  • 辅助策略：保留澄清能力，但严格限制澄清最多不超过两轮
  • 平衡原则：两种方式结合使用，平衡用户体验和需求准确性

行业黑话处理

在意图识别Agent中内置专门处理行业黑话的RAG库，提前收录对应行业的专业词汇、规则，在意图识别阶段就完成专业词汇的准确识别，解决大模型无法理解内部专有词汇的问题。

总结

自研Agent中枢系统的核心价值不在于单点工作流，而在于多工具统一管控——当企业AI工具从1个扩展到N个、使用者从1人扩展到数百人时，需要一个中枢层做身份管理、工具注册、上下文桥接和成本管控。这正是Harness工程在企业级场景的具体落地。

商业化上，B端单点切入比全链路方案更有市场；web coding成为基础办公技能后，企业级Agent OS是确定性机会。AI替换企业节点的初期收益不明显是行业共同难点，需要耐心等待全链路替换完成后的效率涌现。
