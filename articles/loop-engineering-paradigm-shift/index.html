软件工程六十年，工程师的定义一直是"写代码的人"。不管你用什么语言、什么框架、什么架构，最终产出物是代码。Loop Engineering标志着这个定义第一次被动摇：工程师不再直接产出代码，而是设计让AI产出代码的循环系统。 循环，而非提示词，成为AI工程的基本单位。

---

核心结论

AI工程范式不是跳跃式更替，是嵌套式包裹。理解Loop Engineering的位置，需要看清楚这条演进线：

  • Prompt Engineering（2022-2023）：关注"你发送给AI的词语"。核心问题是如何与AI沟通。
  • Context Engineering（2024-2025）：关注"你提供给AI的信息环境"。核心问题是如何为AI提供更好的决策上下文。
  • Harness Engineering（2025）：关注"你为AI搭建的运行环境"。核心问题是如何为AI提供安全的执行沙箱。
  • Loop Engineering（2025-2026）：关注"你设计的让AI自我驱动的循环系统"。核心问题是如何让AI自主完成任务。

这四层是嵌套包含关系，不是替代关系。每层包裹上一层：你设计Loop时，里面仍然需要Harness来提供运行环境，Harness里仍然需要Context Engineering来管理信息，Context里仍然需要好的Prompt。Prompt Engineering的技能没有被淘汰，而是被包裹进更高层抽象中——就像写汇编的技能没有消失，只是被编译器包裹了。

但关键变化在于抽象层级的上移。每次包裹，工程师的注意力就上移一层。Prompt时代，工程师的精力花在措辞上；Context时代，花在信息组装上；Harness时代，花在环境搭建上；Loop时代，花在循环设计上。

核心范式转变：从"人即循环、人是瓶颈"到"人设计循环、人退后"。

在Prompt Engineering阶段，人就是循环——每次AI输出后，人来判断、来提示、来纠偏。AI做得快，人判断得慢，人就是瓶颈。Loop Engineering做的事，是把人从这个循环里抽出来：你设计循环的起止条件和验证逻辑，AI自己在循环里跑。

---

二、"写循环"取代"写代码"

Addy Osmani在奠基性文章中给出了最精确的定义：

"Loop engineering is replacing yourself as the person who prompts the agent. You design the system that does it instead."

——循环工程就是把你这个"提示agent的人"替换掉，你设计那个替你做这件事的系统。

Andrew Ng随后的评价更直白："我们正真正从写代码转向管理意图和规格。"

这两句话合在一起，勾勒出工程师角色的根本变化：

以前，工程师的工作是：理解需求 → 写代码 → 测试 → 修bug。人是执行者，代码是产出物。

现在，工程师的工作变成：设定目标 → 设计工具集 → 设计验证器 → 设计停止条件。人是系统设计者，循环是产出物。

Claude Code的创造者Boris Cherny自己的工作流就是一个经典案例。他设置了一个每30分钟自动轮询社交媒体的routine，AI自动收集反馈、分类摘要、生成issue。他自己不再手动检查反馈——他设计了一个循环来替他做这件事。

注意这个案例的关键点：Boris Cherny没有"写代码"来检查社交媒体反馈。他设计了一个循环——有触发条件（每30分钟）、有执行Agent（Claude Code）、有输出（分类摘要和issue），然后让它自己跑。

这就是范式迁移的实质：工程师的产出物从代码变成了循环。

---

三、一个Loop的五块积木

把Loop Engineering拆到最底层，一个可运行的循环需要五样东西：

1. Goal（目标）：给Agent一个目标，而不是逐步指令。目标的清晰度直接决定循环的质量——模糊的目标必然产生模糊的循环。 2. Tools（工具）：Agent可调用的能力，通过MCP等协议接入。工具集的边界就是Agent能力的边界。 3. Memory（记忆）：跨循环迭代的状态持久化。没有记忆，Agent每次都从零开始，无法积累经验。 4. Verifier（验证器）：确定性测试或CI作为停止条件。验证器的质量就是循环质量的天花板。 5. Stop Condition（停止条件）：防止无限循环消耗token。没有刹车的引擎不是引擎，是炸弹。

这五块积木里，最关键的是验证器和停止条件。前三个（目标、工具、记忆）解决"Agent能不能做"，后两个（验证器、停止条件）解决"循环什么时候停"。

为什么验证器是核心？因为Loop Engineering的本质是让Agent自主纠偏。纠偏的前提是知道"偏了"——这个判断不能靠Agent自己感觉良好，必须有确定性的外部验证。写代码的循环用单元测试做验证器，内容审核的循环用审核准确率做验证器。没有验证器的循环，实质上是在烧token碰运气。

为什么停止条件不可省略？因为Agent Loop的运行成本是线性的——每多跑一轮，就多消耗一轮的token。社区里有人直接称其为"financial nightmare"。一个没有明确停止条件的循环，在财务上等同于一张没有限额的信用卡。

这与SDD（Spec-Driven Development）的关系就清楚了：SDD提供输入（明确的目标和验收标准）和停止条件（规格满足=循环终止）。没有SDD的Loop Engineering就是没有刹车的引擎。

---

四、循环不限于代码

Loop Engineering这个概念从软件开发领域诞生，但它的应用面远不止于此。

任何满足三个条件的场景都可以用Loop：任务可重复、结果可验证、停止条件可定义。

比如说内容审核。一个内容审核Loop的设计：Agent读取待审内容 → 对照审核规则判断 → 输出审核结果 → 验证器抽样复核 → 不通过的案例注入记忆 → 下一轮。整个循环不需要人写任何判断逻辑，人只需要设计审核规则和验证标准。循环跑起来后，人从"审核员"变成了"审核规则设计者"。

比如说广告投放优化。Agent读取投放数据 → 分析ROI → 调整出价策略 → 等待数据回流 → 验证ROI是否提升 → 不满足则回退策略 → 下一轮。这个循环的核心不是"人调广告"，是"人设计调广告的循环"。

再比如一个更贴近日常的场景：知识处理链路。抓取信息 → AI加工结构化 → 人工确认 → 落库 → 定期归档。这本身就是一个Loop——有目标（知识沉淀）、有工具（抓取和加工Agent）、有验证（人工确认环节）、有停止条件（确认通过即落库）。很多人已经在用Loop Engineering的思维做事，只是没有意识到。

实质上，Loop Engineering的真正爆发点不在代码编辑器里，而在所有"重复劳动+可验证结果"的场景里。 代码编辑器只是第一个试验场，因为它天然有测试做验证器、有CI做停止条件。一旦把验证器和停止条件的设计能力迁移到其他领域，Loop的适用面会远大于软件开发。

---

五、三个层级：从玩具到生产

不是所有Loop都一样。按工程复杂度，可以分三个层级：

L1 基础循环：单Agent + 工具调用，一个while循环。这是大多数demo和原型的水平。能跑，但不耐造——没有状态持久化，出错就从头来。

L2 有状态循环：持久化状态、检查点、错误恢复。Agent跑到第47步崩了，能从第46步恢复而不是重头开始。这是生产环境的最低门槛。

L3 多Agent治理循环：多Agent编排、预算断路器、人工审批门。这不是一个循环，是一组循环的协调系统。有预算上限、有降级策略、有人在关键节点把关。

多数实践者目前停留在L1。从L1到L2的跨越，核心是状态管理——LangGraph的检查点机制、记忆持久化方案都在解决"Agent崩了怎么续跑"这个问题。从L2到L3的跨越，核心是治理——当多个Agent同时跑循环，预算控制、优先级调度、人工审批门就成了必需品。

L3的预算断路器是生产环境的硬需求。 它的逻辑很简单：设置硬预算上限（token数/美元/迭代次数），超出时终止循环或降级为人工接管。没有断路器的生产级Loop，迟早会出事——不是因为技术不行，是因为没有成本边界的自主系统一定会撞上成本墙。

六、争议：真范式还是新流行语？

对Loop Engineering的批评集中在三点，逐一回应。

第一，"不过是ReAct的重新包装"。 ReAct（Reason + Act）模式2023年就有了，感知→推理→行动→观察→循环，这跟Loop Engineering有什么区别？

判断是：ReAct是一个Agent内部的推理模式，Loop Engineering是工程师层面的系统设计范式。ReAct解决"Agent怎么想"，Loop Engineering解决"人怎么设计让Agent自主跑的系统"。同一个循环，ReAct是引擎内部的运转方式，Loop Engineering是引擎的设计图纸。 把设计图纸等同于引擎内部的运转方式，是搞混了抽象层级。ReAct是Loop内部的组件之一，但Loop Engineering还包括目标设计、工具编排、验证器构建、预算控制、多Agent协调——这些都不在ReAct的范围内。

第二，Token成本。 Loop Engineering本质是burn more tokens——让Agent多跑几轮，自然多花钱。这个批评是真实的，但不构成范式否定。云计算早期的成本也远高于自建机房，但没有人因此否定云计算的方向。关键在于ROI：如果一个Loop能替代的重复劳动成本高于它消耗的token成本，它就是划算的。随着模型推理成本持续下降，这个盈亏平衡点会持续下移。

第三，"认知投降"风险。 Addy Osmani自己提出了这个担忧：将循环控制权交给Agent后，人类可能逐渐失去对系统行为的理解。这个风险是真实的，但解决方案不是退回手动模式，而是在循环中设计可观测性和人工审批门。L3层级里的Human-in-the-Loop Gate就是为此设计的——在关键步骤前插入人工确认节点，既保留自主性又保留控制权。

这三个争议不构成对Loop Engineering的否定，而是界定了它的边界：需要明确的停止条件、需要成本可控的模型、需要在关键节点保留人类判断。 边界清晰不代表范式不成立，恰恰说明它开始从概念走向工程化。

---

七、工程师的新技能栈

如果工程师的核心产出从代码变成了循环，技能栈就得跟着变。

原来需要精通的：编程语言、框架API、设计模式、调试技巧。

现在需要精通的：目标分解（把模糊需求转化为可验证的Agent目标）、工具设计（为Agent配置合适的工具集）、验证器设计（构建确定性的质量检查机制）、成本控制（设计预算断路器和降级策略）。

不是说编程不重要了——代码仍然是循环内部的执行语言。但代码从"最终产出物"变成了"中间产物"，工程师的注意力从"怎么写好这段代码"上移到"怎么设计好这个循环"。

实质上，这跟每一次抽象层级上移的规律一样。 汇编时代的人觉得高级语言是"偷懒"，C时代的人觉得Python是"不够硬核"。每一次抽象上移，都有一批旧技能被包裹进底层，同时一批新技能在新的抽象层上涌现。Loop Engineering是这条规律在AI工程领域的最新一环。

---

收束

2026年的工程师面对的问题，已经不是"怎么写好这段代码"，而是"怎么设计一个让AI替我写代码的循环，并且确保它知道什么时候该停"。

这个转变的深层含义是：工程师的价值，从"亲手产出"变成了"设计产出系统"。 以前你的价值体现在你写的代码里，现在你的价值体现在你设计的循环里——循环的质量决定了AI产出的质量。

Loop Engineering会不会被更高层的抽象包裹？大概率会。就像Prompt Engineering被Context Engineering包裹，Context被Harness包裹，Harness被Loop包裹一样。但在2026年，Loop就是当前的最前沿。

设计循环的人，取代了写代码的人。这个变化已经发生了。
