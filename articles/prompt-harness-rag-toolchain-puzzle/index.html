核心结论

大部分人以为提示词就写在聊天框里。不是的。

完整的提示词体系，是三层拼图叠加的结果。

第一层，内置系统指令。这部分在大模型内部，你看不到也改不了。它是底层的安全策略、格式约束、行为规范。

第二层，企业云端策略。如果你的公司有企业配置，这部分提示词会从云端注入，不在你的本地机器上。

第三层，你的Skill和Instruction。你写的instruction、你创建的SKILL.md，这些会被加载进系统提示词。

也就是说——你以为自己在本地写了条prompt，其实它已经跟另外两层拼在一起，变成了完整的系统提示词。

这里有一个关键细节：系统提示词一旦变动，所有缓存失效，推理成本变成原来的10倍。 这也是为什么你要尽量保持系统提示词稳定——不是因为你写的有问题，是因为改一次，整个缓存层要重建。

二、Skill是提示词的"稳定器"

知道第一点之后，Skill的作用就好理解了。

Skill本质上是一个"提示词稳定器"。 你把最核心的规则、配色、格式要求写进SKILL.md，它就被提前加载到系统提示词里，成为稳定的一部分。之后你每次对话，这些内容不会被重复翻译、重复拼接、重复缓存——它就是"底色"。

除了SKILL.md，Skill还可以带scripts文件夹（可执行脚本）、references文件夹（参考文档，需要时才读）、assets文件夹（模板、字体、图片等）。这就是"渐进式披露"：不是一次性把所有东西塞进上下文，而是按需加载。

一句话总结：Skill的做法是"提前声明，稳定加载，按需展开"。目的只有一个——别让系统提示词乱动。

三、Harness的核：Agent就是"问→做→问"

最简单的Agent，核心就是一个无限循环。

用户给指令 → 模型判断：需不需要用工具？
  ├── 不需要 → 直接输出答案
  └── 需要 → 调用工具 → 拿结果 → 回到"继续想"

就这么简单。你每用一次AI编程工具，背后都是这个循环在跑。它"想"一下，发现需要一个文件的内容，调Read工具，读完了继续"想"——可能还需要改一下文件，调Edit；可能还需要跑个命令，调Bash。直到它"想"完了，不需要工具了，就把结果输出给你。

在这个循环上，逐渐长出了更多东西。为了知道"我在哪一步、还剩哪些事没做"，加了Todo管理。为了让一个Agent不爆炸，加了子Agent模式——父Agent负责协调分配，子Agent负责具体执行。一个任务太复杂，就拆成几个小的，各自跑各自的循环。

四、子Agent和Skill是两种不同的"分工"

这两样东西经常被放在一起聊，但逻辑完全不同。

子Agent是任务拆解。 一件事太大了，一个脑子装不下，拆成几个子任务，每个分配一个子Agent去跑。父Agent不干活，只协调——它告诉你"你去做这个，你去做那个"，最后汇总结果。

Skill是能力注入。 不是拆任务，是给Agent配了"外挂知识库"。每次做这类事之前，先读一下这份指南。SKILL.md就是那本"碰到这类任务该怎么干"的手册。

打个比方：子Agent是"多招几个人"，Skill是"给每个人发一本操作手册"。一个是横向拆解，一个是纵向赋能。

五、RAG：让模型读到它不知道的东西

模型的知识是冻结的。训练完那天，它的"脑子"就停在那一刻了。 RAG做的事情就是：在推理的时候，把新知识"塞"进它的上下文里。

完整流程分六步：

数据加载 → 把PDF、网页、数据库里的东西捞出来。

文本分块 → 把长文档切成一段一段，每块之间5%-10%的重叠，避免硬切切断语义。

向量化 → 用embedding模型把文本块转成向量。

向量索引 → 存进向量数据库。

检索 → 用户问一个问题，系统去向量库找最相关的片段。

拼入上下文 → 把找到的片段和用户问题一起塞给模型。

产品经理和算法看RAG，看的是两个不同的东西。 产品经理关注前半段——数据怎么加载、文本怎么切、块大小调多少、重叠率设多大。算法关注后半段——embedding模型选哪个、向量索引怎么建、检索策略怎么设计。

切得太碎，语义丢了；切得太大，检索精度差。分块策略是RAG效果的核心变量之一。

六、三者关系：同一问题的三个维度

这三个话题——提示词、Harness、RAG——本质上是同一个问题：怎么让AI干得更好。

| 维度 | 解决的问题 | 核心机制 | |------|-----------|---------| | 提示词 | 你怎么告诉它 | 三层拼图，保持系统提示词稳定 | | Harness | 它怎么干 | 问→做→问的无限循环 | | RAG | 它不知道的怎么让它知道 | 检索→拼入上下文 |

你每次写提示词、每次配Skill、每次设计RAG流程，本质上都在参与AI的工具链设计。不是工程师才能干的事。
