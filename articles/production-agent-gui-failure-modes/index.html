核心结论

三类Agent定位对比：

  • 纯Code Agent：运行在封闭纯文本环境，易扩展但场景受限
  • 具身智能：运行在物理世界，能力潜力大但数据获取与硬件成本极高
  • GUI Agent：结合两者特点，运行在电脑/手机的封闭虚拟环境，可扩展同时需要感知能力，处于研究的甜点位置

GUI Agent的研究成果可同时反哺前两类Agent。

当前主流GUI Agent都基于React框架，分为观察（Observe）、思考推理（Reasoning）、动作执行（Action）三步，每一步完成后进入新的状态循环，直到任务完成。

二、GUI Agent的三种常见失败模式

1. 不看图失败

模型依赖历史文本记录做决策，不参考当前界面图片。本质原因是训练数据存在大量捷径——很多任务仅通过文本即可得到答案，模型自然学到不看图的行为模式。

解法：SFT阶段数据清洗，去除不看图的错误pattern。

2. COT漂移问题

模型仅在初始阶段将图片转文本，后续推理全部在文本空间完成，不会持续检查图片状态。同时GUI是人工设计图像，元素排版带有语义引导，和自然图像读图逻辑差异大。

解法：针对性训练，当前在图像空间做推理仍是开放问题。

3. 不可信历史问题

长轨迹执行中只能保留1-2张最新图片，历史操作以文本形式记录，但实际执行动作和预期动作可能存在偏差——点击位置因广告加载偏移、模型点击位置偏差——导致记录的历史和真实状态不一致。

解法：大规模rollout加在线RL训练。

核心判断：GUI Agent的三种失败模式都不是"模型不够聪明"的问题，而是训练数据和训练方式的问题。解法都指向同一个方向——用更多真实执行数据训练模型。

三、GUI Agent迭代方案

迭代逻辑：大规模rollout收集带失败标注的轨迹数据 → SFT阶段清洗错误pattern → RL阶段激发正确推理行为 → 自研reward system对每步执行打分 → 迭代式更新。

Code+GUI混合模式是未来方向：合版后的模型可结合两种能力——买火车票任务可先调用搜索获取关键信息缩短执行步骤，能完成更多之前无法完成的复杂任务。未来可覆盖CAD等专业软件。

四、需求分析Engine：四层架构

需求分析Engine解决的核心问题：产品方提出的需求往往不够成熟，需要工具梳理成完整规范的产物。

核心架构分层

| 层级 | 职责 | 说明 | |------|------|------| | 准备层 | 补充需求上下文 | 从知识库匹配已有同类需求经验 | | 编排层 | 推理路由 | 根据需求复杂度选择不同处理流程，最终生成PRD文档 | | 能力层 | 注入领域skills | 支持不同领域的专属skills，适配不同类型需求 | | 数据层 | 入库分析 | 步骤分析，优化路由逻辑保障系统稳定性 |

主进程+子Agent分层

主进程只保留清晰简短的上下文，将不同子任务分配给对应子Agent（如合规检查、安全校验），解决上下文长度爆炸问题。

关键设计：将事件流和工具调用完全剥离，后续叠加新工具、新Agent不会影响主线流程，迭代更清晰。

安全与稳定性

  • 对工具调用做严格权限控制，防范恶意代码攻击
  • 增加trust节点记录，做成本优化和异常检测，防止模型无限思考占满上下文
  • 最终产物输出后额外做一次自我检测，不达标则自动修订，直到结果符合要求

核心判断：需求分析Engine的本质不是"让AI写PRD"，是"让AI把模糊需求约束成可执行spec"——这跟Spec驱动开发是同一个问题。

五、需求分析Engine的工程优化

企业知识库构建

将git仓库、文档库、飞书等不同来源的知识绑定到Agent，统一扫描分类上传。

生成结果评估

每个节点挂载子Agent做评审，用预训练的优质示例做参考，同时采用多模型交叉判断，提升结果质量和稳定性。

降低部署成本

采用薄记忆层方案，针对不同行业场景沉淀动态的薄记忆，自动生成适配的workflow，快速完成部署。

隐性业务知识挖掘

先用AI解读切片生成项目代码文档，再通过多轮迭代逐步补全，配合skill方法和AI编码工具，整理为规范化图谱供AI读取。

六、Loop Engineering的圆桌讨论

观点一（Agent工程师）：Agent需要完成自闭环，用户希望Agent可以自行验证迭代成果，Loop Engineering符合当前Agent发展需求。

观点二（需求分析Engine开发者）：Loop本质是让模型自我修正结果，属于原有工程概念的延伸，不是爆炸性的全新概念，对效率的提升没有特别大的突破。

观点三（产品视角）：Loop Engineering覆盖模型、工程、产品三个层面，产品迭代本身就是Loop，它的价值是完成了破圈——让产品、业务同学也能理解工程化迭代的思路。

核心判断：Loop Engineering的价值不在于技术突破，在于概念破圈——让非工程角色也能理解"迭代→验证→修正"的闭环思维。

七、Token消耗增长下的模型选型

选型思路

优先保证任务完成效果：复杂任务用顶尖大模型，简单任务用轻量化模型降低成本。

企业级落地采用模型池分层方案：核心推理、最终输出使用高价优质模型，基础步骤使用低成本模型。

实际成本优化经验

有团队最初用GPT开发需求分析Engine，每天消耗5000美元。后续通过按环节切分模型——核心环节用高级模型，子步骤和评测用国产开源模型——结合上下文管理，有效降低了整体成本。

核心判断：模型选型不是"选一个最好的模型"，是"把不同环节匹配到性价比最优的模型"。成本优化的关键在分层，不在压价。

八、模型好坏的衡量维度

抛开benchmark，主要依靠个人使用体感，需要实际上手测试才能判断模型是否符合需求。

不同模型擅长方向不同：写作适合某些模型，代码适合另一些模型，样式优化又不同。需要根据任务类型选择，没有万能模型。
